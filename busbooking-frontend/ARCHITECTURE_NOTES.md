# Заметки по архитектуре: Vite/React ERP vs наш Next.js busbooking-frontend

## 1. Структура `src/` в ERP-проекте (Vite + React) и её аналог у нас

| Папка/файл в ERP | Роль в ERP                                                                                                                                          | Аналог в нашем Next.js проекте                                                                                                                                                 |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `__generated__`  | Автосгенерированный код — типы/клиент API (из OpenAPI/Swagger или GraphQL codegen)                                                                  | У нас Swagger уже есть (`/swagger`) — можно генерировать TS-типы из него в `types/__generated__` вместо ручного дублирования VM-интерфейсов                                    |
| `api`            | Слой HTTP-клиента: обёртки над fetch/axios, base URL, интерцепторы (refresh-токен, обработка 401)                                                   | `lib/api.ts` — общий клиент с `NEXT_PUBLIC_API_BASE_URL`, туда же логика refresh-интерцептора для JWT                                                                          |
| `auth`           | Всё про authentication flow: контекст/провайдер текущего юзера, проверка permissions, guard-компоненты для защищённых роутов                        | `auth/` — `AuthProvider`/`useAuth` (client-side, т.к. в App Router нет server-side "текущего юзера" без сессии) — обёртка над JWT в localStorage/cookie, редиректы на `/login` |
| `components`     | Общие переиспользуемые UI-компоненты (кнопки, таблицы, модалки), не привязанные к конкретной странице                                               | `components/ui/` — то же самое                                                                                                                                                 |
| `hooks`          | Общие хуки, не привязанные к одной странице (в отличие от, например, `useCategoriesPageVM`, который живёт внутри `pages/CategoriesPage`)            | `hooks/useDebounce.ts`, `hooks/useMediaQuery.ts` и т.п. — только общие хуки, не VM-хуки страниц                                                                                |
| `lib`            | Утилитарные библиотеки/обёртки над сторонними пакетами (конфиг TanStack Query, форматтеры дат)                                                      | `lib/queryClient.ts`, `lib/format.ts`                                                                                                                                          |
| `pages`          | Сами страницы; в Vite-проекте — ручной роутинг (`router.tsx` описывает пути → компонент)                                                            | В Next.js этой папки как отдельной сущности нет — её роль берёт на себя `app/` (файловый роутинг сам создаёт маршруты по структуре папок)                                      |
| `stores`         | Глобальное состояние вне React Query (скорее всего Zustand/Redux) — то, что не привязано к серверным данным (например, UI-state: открыт ли сайдбар) | Нам, скорее всего, не понадобится вовсе на старте — TanStack Query покрывает серверные данные, а локального UI-состояния так мало, что глобальный store избыточен              |
| `types`          | Общие TS-типы, не сгенерированные автоматически (ручные интерфейсы)                                                                                 | `types/` — то, что не покрыто кодогенерацией из Swagger                                                                                                                        |
| `utils`          | Чистые функции без побочных эффектов (форматирование, валидация)                                                                                    | То же самое                                                                                                                                                                    |

## 2. Почему Vite (чистый CSR) не подходит для публичных страниц

Минусы для публичного сайта на чистом Vite/CSR:

- **SEO** — поисковый бот видит пустой `<div id="root">`, пока не выполнится весь JS-бандл. Google в целом умеет дожидаться рендера, но делает это медленнее и хуже ранжирует; другие боты (Bing, соцсети для превью ссылок) часто вообще не выполняют JS.
- **Первая отрисовка (FCP/LCP)** — пользователь видит пустой экран, пока не подгрузится и не выполнится весь бандл.
- **Нет автоматического пререндера** — добавляется вручную (доп. пакеты, доп. конфиг).

## 3. Next.js: рендеринг решается на уровне каждой страницы

Next.js даёт выбор для каждой страницы отдельно, где и когда она рендерится:

- **SSG (Static Site Generation)** — HTML собирается один раз на этапе сборки (`next build`), отдаётся как статический файл. Для Home/About Us — идеально: контент не меняется, отдаётся мгновенно, кэшируется на CDN.
- **ISR (Incremental Static Regeneration)** — тот же статический HTML, но пересобирается в фоне раз в N секунд (`revalidate: 3600`). Для каталога маршрутов — цены/направления обновляются раз в час, а не при каждом запросе.
- **SSR (Server-Side Rendering)** — HTML собирается на сервере при каждом запросе (`cache: 'no-store'`). Нужен, когда контент зависит от конкретного запроса (например, персонализация).
- **CSR** — тоже доступен через `"use client"` + обычные хуки, ровно как в Vite — для интерактивных кусков (форма логина, выбор места), где SEO не важен.

### Ключевая идея

В Next.js публичные страницы прилетают в браузер уже готовым HTML (бот и пользователь сразу видят контент), а интерактивные защищённые страницы (после логина) ведут себя как обычная SPA — то есть оба режима сосуществуют в одном проекте, выбор делается по каждой странице отдельно.

## 4. Итог для нашего кейса (BusBooking)

- Home / About Us / каталог маршрутов — публичные страницы, их должен находить Google и мгновенно видеть посетитель → нужен **SSG/ISR** → это ровно то, чего Vite не даёт из коробки, а Next.js даёт нативно.
- Выбор Next.js для этого проекта архитектурно оправдан, и его стоит оставить.

## 5. `useState` и `useEffect` — на примере `auth/AuthProvider.tsx`

### 5.1 `useState` — память компонента

```ts
const [user, setUser] = useState<AuthUser | null>(null);
```

- `user` — текущее значение состояния (в момент первого рендера — `null`).
- `setUser` — единственный способ его поменять.
- `null` в скобках — начальное значение (используется только при самом первом рендере компонента).

**Почему нельзя `user = ...`.** React не «наблюдает» за обычными JS-переменными — он не узнает, что что-то изменилось, и не перерисует компонент. Вызов `setUser(...)` делает две вещи: (1) сохраняет новое значение во внутреннем хранилище React, привязанном к этому конкретному экземпляру компонента, и (2) ставит компонент в очередь на повторный рендер. При следующем рендере `useState` вернёт уже новое значение.

**Batching.** Несколько вызовов `set...` подряд в одном обработчике события React объединяет в один re-render, а не делает N отдельных:

```ts
setUser(decoded);
setIsLoading(false); // не вызовет второй лишний рендер — обе обновятся вместе
```

**Lazy-инициализация.** Если начальное значение дорого вычислять, можно передать функцию вместо значения — она выполнится один раз, только при первом рендере:

```ts
useState(() => expensiveComputation());
```

Мы этим приёмом в `AuthProvider` **не** пользуемся сознательно — см. 5.3, почему.

### 5.2 `useEffect` — побочные эффекты

```ts
useEffect(() => {
  const tokens = getTokens();
  if (tokens) {
    setUser(decodeAccessToken(tokens.accessToken));
  }
  setIsLoading(false);
}, []);
```

`useEffect` — способ сказать React: «после того как отрисуешь этот компонент на экране, выполни вот этот код». Это единственное штатное место для всего, что не является чистым вычислением JSX: чтение `localStorage`, HTTP-запрос, подписка на событие браузера (`window.addEventListener`), таймеры.

**Второй аргумент — массив зависимостей.** Он решает, _когда_ эффект перезапускается:

| Зависимости                              | Когда срабатывает                                                          |
| ---------------------------------------- | -------------------------------------------------------------------------- |
| Не указаны вовсе (нет второго аргумента) | После **каждого** рендера — почти всегда ошибка                            |
| `[]` — пустой массив                     | Один раз, сразу после первого рендера (аналог `componentDidMount`)         |
| `[pathname]`                             | При первом рендере, и затем повторно каждый раз, когда `pathname` меняется |

В `AuthProvider` — `[]`, потому что чтение токена из `localStorage` должно случиться ровно один раз, при монтировании провайдера, а не при каждом изменении `user` (что было бы бесконечным циклом: эффект меняет `user` → рендер → эффект снова видит смену зависимости → и т.д.).

**Функция очистки (cleanup).** Если из эффекта вернуть функцию, React вызовет её перед следующим запуском эффекта и перед размонтированием компонента — используется для отписки от событий/таймеров, чтобы не текла память:

```ts
useEffect(() => {
  const onScroll = () => setScrolled(window.scrollY > 50);
  window.addEventListener("scroll", onScroll);
  return () => window.removeEventListener("scroll", onScroll); // cleanup
}, []);
```

(Ровно так устроен `Header.tsx` — там `useEffect` подписывается на скролл и отписывается при размонтировании.)

### 5.3 Почему именно `useEffect`, а не чтение `localStorage` прямо в теле компонента

Next.js часть рендера выполняет **на сервере** (SSR/SSG) — и на сервере объекта `localStorage` физически не существует (это API браузера). Если написать:

```ts
const tokens = getTokens(); // прямо в теле компонента, без useEffect
```

— это упадёт с ошибкой при серверном рендере, потому что `localStorage` там `undefined`.

`useEffect` гарантированно выполняется **только в браузере**, и только **после** того, как React уже отрисовал компонент — то есть уже после того, как сервер закончил свою часть работы. Поэтому:

1. Сервер рендерит страницу, ничего не зная о `localStorage` → `user = null` (значение по умолчанию из `useState`).
2. Браузер получает готовый HTML и «оживляет» его (hydration) — на этом этапе HTML на сервере и первый рендер на клиенте обязаны совпадать, иначе React ругается на hydration mismatch.
3. Сразу после этого срабатывает `useEffect` → читает `localStorage` → если токен есть, `setUser(...)` → React перерисовывает компонент уже с реальным пользователем (например, в хедере появляется кнопка «Sign out»).

Именно поэтому в `AuthProvider` **нельзя** заменить эту связку на lazy-инициализацию `useState(() => getTokens() && ...)` — она бы выполнилась синхронно во время первого клиентского рендера, до/во время hydration, и могла бы дать другой результат, чем то, что уже отрисовал сервер (`null`) → React заметит расхождение и выдаст предупреждение о hydration mismatch, либо тихо покажет неконсистентный UI на долю секунды.

### 5.4 Замеченный lint-warning: `react-hooks/set-state-in-effect`

ESLint-правило ругается на `setUser(...)` внутри `useEffect` — формально это «вызов `setState` синхронно внутри эффекта», что действительно может каскадно триггерить лишние рендеры, если использовать не по назначению. Но именно наш кейс (переход из «состояние неизвестно» → «состояние прочитано из внешнего источника, доступного только в браузере») — ровно то, для чего `useEffect` официально существует, и альтернативы без hydration-проблем (см. 5.3) в App Router нет.

`Header.tsx` изначально нарушал то же правило по другой причине: эффект синхронизировал локальное состояние `menuOpen` с изменением `pathname` (закрывал мобильное меню при переходе на другую страницу) — это уже классический анти-паттерн «подгонка state под изменившийся проп через эффект», а не синхронизация с внешней системой, как в `AuthProvider`. React рекомендует делать это прямо во время рендера (см. https://react.dev/learn/you-might-not-need-an-effect#adjusting-some-state-when-a-prop-changes), поэтому эффект заменён на паттерн сравнения "текущий проп vs сохранённый в state предыдущий":

```ts
const [prevPathname, setPrevPathname] = useState(pathname);
if (pathname !== prevPathname) {
  setPrevPathname(pathname);
  setMenuOpen(false);
}
```

Вызов `setState` прямо в теле компонента (не в эффекте) допустим в React и не создаёт лишнего цикла рендер → эффект → рендер — React обнаруживает такой вызов во время текущего рендера и применяет его перед показом на экране, без промежуточного кадра.

## 6. npm vs pnpm

Оба — менеджеры пакетов Node.js с одинаковым CLI-интерфейсом (`install`, `run`, `build` и т.д.), но с разной механикой хранения зависимостей на диске.

|                          | npm                                                                                                                                                           | pnpm                                                                                                                                                              |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Хранение зависимостей    | Копирует каждый пакет в `node_modules` каждого проекта отдельно (с npm@7+ частично дедуплицирует через плоскую структуру, но физические копии всё равно есть) | Хранит один физический экземпляр пакета в глобальном content-addressable сторе (`~/.pnpm-store`), а в `node_modules` проекта — жёсткие ссылки (hardlinks) на него |
| Место на диске           | Больше — при N проектах с одинаковой зависимостью она физически хранится N раз                                                                                | Меньше — одна физическая копия на все проекты на машине                                                                                                           |
| Скорость установки       | Медленнее, особенно при повторных установках "с нуля" (`node_modules` удалён)                                                                                 | Быстрее за счёт переиспользования стора — копирование не требуется, только линковка                                                                               |
| Строгость `node_modules` | "Плоская" структура — пакет может импортировать чужую транзитивную зависимость, даже не указав её в своём `package.json` (phantom dependencies)               | По умолчанию строгая, вложенная структура — виден только явно указанный в `package.json` пакет, phantom dependencies физически недоступны для импорта             |
| Монорепозитории          | Есть workspaces, но менее эффективны для больших монорепо                                                                                                     | Изначально проектировался с прицелом на монорепо — `pnpm-workspace.yaml`, эффективнее по месту и скорости на монорепо                                             |
| Lock-файл                | `package-lock.json`                                                                                                                                           | `pnpm-lock.yaml`                                                                                                                                                  |

**В нашем проекте:** сейчас используется `npm` (см. `package-lock.json` в корне `busbooking-frontend/`) — стандартный выбор, ничего менять не требуется, просто фиксирую разницу для справки. Если возникнет необходимость ускорить установки в CI или начать монорепо (например, объединить `BusBooking` backend и `busbooking-frontend` под одним workspace-менеджером для npm-скриптов), pnpm — разумная альтернатива для рассмотрения.

## 7. `useSearchParams()` требует `<Suspense>` при статической сборке

Обнаружено на `app/login/page.tsx` → `useLoginPageVM.ts` при `npm run build`:

```
⨯ useSearchParams() should be wrapped in a suspense boundary at page "/login".
```

**Причина.** `useSearchParams()` читает query-параметры текущего URL (у нас — `redirectTo`, чтобы после логина вернуть пользователя туда, откуда его отправили на `/login`). Значение query-параметров в принципе неизвестно на этапе `next build` — они появляются только когда браузер делает конкретный запрос с конкретным URL. Поэтому Next.js не может включить их в статически сгенерированный (prerendered) HTML страницы и вынужден рендерить эту часть дерева уже на клиенте (CSR bailout).

Чтобы пререндер остальной части страницы (статичный HTML — заголовок, форма, ссылки) не блокировался этим клиентским кусочком, React/Next.js требует явно объявить границу: какой fallback показать, пока `useSearchParams()` ещё не разрешился на клиенте. Для этого служит `<Suspense>` — без него сборка не знает, что подставить вместо ещё не готового поддерева, и падает с ошибкой вместо того, чтобы молча отрендерить пустоту.

**Исправление (`app/login/page.tsx`):** компонент, вызывающий `useLoginPageVM()` (а через него — `useSearchParams()`), вынесен в отдельный `LoginForm`, а `LoginPage` оборачивает его в `<Suspense>`:

```tsx
export default function LoginPage() {
  return (
    <Suspense>
      <LoginForm />
    </Suspense>
  );
}

function LoginForm() {
  const { email, setEmail /* ... */ } = useLoginPageVM();
  // ...JSX формы логина
}
```

Без `fallback` у `<Suspense>` подойдёт для этого случая: сама форма логина рендерится мгновенно на клиенте, разница в один кадр не заметна визуально — в отличие, например, от медленно грузящихся данных с сервера, где `fallback` (скелетон/спиннер) обязателен для UX.

## 8. `getRoutes()` больше не роняет сборку, если API недоступен

`app/page.tsx` — серверный компонент, который на этапе `next build` вызывает `getRoutes()` (`api/routes.ts`), чтобы сгенерировать статический HTML для `/` (ISR, `revalidate: 3600`, см. раздел 3 и 7 выше). Если бэкенд в этот момент не запущен, `fetch` внутри `apiFetch` падает с `ECONNREFUSED`, и весь `next build` завершается ошибкой — даже если ни один код страницы не сломан.

**Исправление (`api/routes.ts`):**

```ts
export async function getRoutes(): Promise<RouteSummary[]> {
  try {
    return await apiFetch<RouteSummary[]>("/api/routes", {
      next: { revalidate: 3600 },
    });
  } catch (error) {
    console.error("getRoutes: falling back to an empty list", error);
    return [];
  }
}
```

Это безопасно, потому что `SearchWidget` и `RouteExplorer` (единственные потребители `routes`) уже корректно обрабатывают пустой массив — `RouteExplorer` даже показывает готовое сообщение `"No routes for this day yet."` вместо падения. Проверено сборкой в обоих режимах: с поднятым бэкендом (`dotnet run` в `BusBooking/`, `NEXT_PUBLIC_API_BASE_URL=http://localhost:5100` из `.env.local`) и без него — оба раза `npm run build` проходит, разница только в том, попадают ли реальные маршруты в статический HTML или страница уйдёт в пустое состояние до первой успешной ISR-регенерации в проде.

## 9. «Обновляется только часть страницы» — четыре разных технологии под одним вопросом

Формулировка неоднозначна — под ней скрывается как минимум четыре разных механизма на разных уровнях стека. Ключевое различие между ними: **что именно** «частично» — рендеринг на сервере (какой HTML собирается), гидрация в браузере (какой JS выполняется) или обновление уже отрисованного DOM (что перерисовывается на экране после взаимодействия).

### 9.1 Islands Architecture — то, что имелось в виду

Архитектурный паттерн (Astro, Fresh, отчасти Qwik): страница по умолчанию — это чистый статический HTML без JS вообще. Интерактивные виджеты («острова» — например, форма поиска, карусель) явно помечаются автором и гидрируются **независимо друг от друга**, каждый своим изолированным кусочком JS. Если на странице 5 островов, а интерактивен нужен только один — в браузер грузится JS только для него, остальные 4 острова так и остаются мёртвой разметкой без обработчиков событий.

**Ключевое отличие от обычного React/Next.js:** в «обычном» React-приложении (в т.ч. в App Router) единица гидрации — это всё дерево клиентских компонентов целиком (граница `"use client"` гидрируется как одно целое вместе со всеми детьми); в Islands — единица гидрации это конкретный виджет, соседние виджеты друг про друга не знают и гидрируются по отдельности, вплоть до разных фреймворков на одной странице (остров на React, остров на Svelte — обычная практика в Astro).

**У нас в проекте:** Next.js App Router архитектурно не Islands (у нас модель Server/Client Components, см. ниже), но по духу ближе всего к нему — компоненты вроде `Header.tsx` или `SearchWidget` объявлены `"use client"` и гидрируются отдельно от полностью статичных `HeroSection`/`WhyPometco`/`ContactSection`, которые в браузер вообще не отправляют JS для рендеринга разметки. Формальной границы «остров = независимая единица гидрации без общего родительского клиентского дерева» в Next.js нет — ближайший официальный термин для нашего стека это Partial Prerendering (см. 9.3).

### 9.2 React reconciliation (virtual DOM diffing) — обновление внутри уже смонтированного дерева

Это не про то, какой HTML/JS отдаётся изначально, а про то, что происходит **после**, когда меняется state внутри уже работающего в браузере React-приложения. При `setMenuOpen(true)` в `Header.tsx` (раздел 5.4) React не перерисовывает всю страницу и не трогает DOM `HeroSection` рядом — он сравнивает новое виртуальное дерево `Header` со старым и точечно обновляет в реальном DOM только те узлы, которые реально изменились (в данном случае — `class` на `<header>` и появление/исчезновение `<nav>` мобильного меню).

Это работает всегда, в любом React-приложении, независимо от того, Islands оно, SSR или чистый CSR — это фундамент, а не отдельная опциональная технология.

### 9.3 Partial Prerendering (PPR) — ближайший официальный аналог Islands в Next.js

Экспериментальная (на момент актуальности этой заметки) фича Next.js App Router: **одна и та же страница** одновременно содержит статически пререндеренную оболочку (мгновенно отдаётся из CDN/кэша, без ожидания сервера) и динамические «дыры» — куски, обёрнутые в `<Suspense>`, которые стримятся с сервера отдельно, уже после того как статическая оболочка показана пользователю.

Разница с Islands: единица «острова» в PPR — это граница `<Suspense>` вокруг серверного рендеринга (что показать пользователю, пока сервер досчитывает динамические данные), а не единица клиентской гидрации JS. Наш `app/login/page.tsx` (раздел 7) уже использует ровно этот примитив (`<Suspense>` вокруг `LoginForm`), просто не как оптимизацию производительности, а как обязательное требование сборки для `useSearchParams()`.

### 9.4 Streaming SSR + Suspense — то же самое, но осознанно как приём для скорости

Формально то же самое, что и 9.3 на уровне API (`<Suspense>`), но с другим намерением: не «удовлетворить требование сборки», а сознательно разбить медленную страницу на быстрый скелет + несколько независимо подгружающихся кусков, каждый со своим `fallback` (спиннер/скелетон), чтобы пользователь увидел хоть что-то мгновенно, а не ждал самого медленного fetch на странице.

**Применено на `app/routes/page.tsx`.** Страница была одним `async`-компонентом: `await Promise.all([getRoutes(), getRoutesVM(...)])` блокировал вообще весь HTML, включая заголовок и форму поиска, до завершения обоих запросов. Теперь страница синхронная, а два независимых куска вынесены в отдельные `async`-компоненты под своими `<Suspense>`:

```tsx
export default function RoutesPage({ searchParams }) {
  return (
    <>
      <section>
        {/* заголовок рендерится сразу, без ожидания */}
        <Suspense fallback={<SearchWidgetSkeleton />}>
          <SearchWidgetSection />
        </Suspense>
      </section>

      <Suspense fallback={<RoutesResultsSkeleton />}>
        <RoutesResultsSection searchParams={searchParams} />
      </Suspense>
    </>
  );
}
```

`SearchWidgetSection` ждёт только `getRoutes()` (кэшируется на 1 час, обычно быстро), `RoutesResultsSection` — более тяжёлую пару `getRoutes()` + `getRoutesVM()` с фильтрацией по `searchParams`. Каждый кусок стримится в браузер по готовности, независимо от соседнего — HTML-заголовок долетает до пользователя раньше, чем досчитается список маршрутов.

### 9.5 Даунсайды на SEO — когда они реально есть

Формулировка возникла в контексте вопроса «а нет ли у Suspense-стриминга минусов для SEO». Ответ: у _серверного_ `<Suspense>` (9.3–9.4) — нет, потому что контент всё равно рендерится на сервере и приходит в составе того же HTTP-ответа, просто не одним блоком, а потоком (chunked transfer encoding); поисковый бот получает полный HTML после завершения ответа, ничего не «уезжает» на клиент.

Даунсайд появляется в другом сценарии — если контент вообще не рендерится на сервере, а подгружается уже в браузере через `useEffect` + `fetch`/`setState` (чистый CSR, как в примере из раздела 5.3 с `localStorage`). Тогда первый HTML, который видит бот, содержит пустое место вместо контента:

- Google-бот в целом умеет дожидаться выполнения JS перед индексацией, но делает это медленнее и с более низким приоритетом рендеринга, чем для готового HTML.
- Другие боты — Bing, парсеры превью ссылок в соцсетях (Telegram/Facebook/Viber-превью, актуально для нашего футера) — как правило JS не выполняют вообще и увидят пустую страницу.

Практический вывод для проекта: публичные страницы (`/`, `/routes`, `/about`) обязаны получать данные через серверные компоненты (`async function Page()` / серверный `fetch`), а не через клиентский `useEffect` — именно поэтому `getRoutes()`/`getRoutesVM()` вызываются в серверных компонентах, а не в хуке на клиенте.

## 10. Термины для оптимизации бэкенда (ASP.NET Core + EF Core + Postgres)

Пункты из раздела «Оптимизация и скорость загрузки» в `ROADMAP.md` (Фаза 10.5) — расшифровка на пальцах, что каждый термин значит и почему он важен именно для нашего API.

### 10.1 N+1-запросы

Ситуация: код запрашивает список из N сущностей одним запросом, а затем для _каждой_ из них отдельно ходит в базу за связанными данными — итого 1 + N запросов к БД вместо одного. Например: получить все `Trip`, а потом в цикле для каждого `Trip` отдельным запросом достать его `Route`. При 50 рейсах это 51 обращение к Postgres вместо одного JOIN.

**Лечится** через EF Core `.Include()` (жадная загрузка связанных сущностей одним SQL-запросом с JOIN) — вместо того, чтобы позволять EF лениво догружать связанные данные по одной сущности за раз при каждом обращении к навигационному свойству.

### 10.2 Индексы БД

Индекс — это дополнительная структура данных поверх колонки (аналог оглавления в книге), которую база использует, чтобы находить строки без полного перебора таблицы (full table scan). Postgres автоматически создаёт индекс только для первичного ключа (`Id`). Для любой другой колонки, по которой часто идёт `WHERE`/`JOIN` (например `SeatDetail.TripId` — «дай все места этого рейса», `Trip.RouteId` — «дай все рейсы этого маршрута»), без явного индекса Postgres будет сканировать всю таблицу целиком на каждый такой запрос. Пока таблицы маленькие (десятки-сотни строк) разницы не видно; по мере роста данных (тысячи бронирований) — это первое, что начинает тормозить.

### 10.3 `AsNoTracking()`

По умолчанию EF Core «отслеживает» (tracking) каждую загруженную сущность — держит в памяти снимок её исходного состояния, чтобы потом суметь понять, что изменилось, и сгенерировать `UPDATE` при `SaveChanges()`. Это нужно только для сущностей, которые дальше будут _изменяться_ в рамках того же запроса. Для чисто читающих эндпоинтов (`GET /api/trips`, `GET /api/routes` — отдать данные и забыть) это чистые накладные расходы CPU/памяти без пользы. `.AsNoTracking()` в LINQ-запросе отключает эту бухгалтерию для конкретного запроса.

### 10.4 Кэширование ответов

Вместо того чтобы на _каждый_ HTTP-запрос заново ходить в БД и пересчитывать ответ, часто запрашиваемые и редко меняющиеся данные (например, каталог маршрутов — меняется от силы несколько раз в день) можно один раз посчитать и отдавать из памяти следующим N запросам, пока не истечёт время жизни кэша или данные не изменятся. В ASP.NET Core это `AddOutputCache`/`AddResponseCaching` (аналог по духу тому, что `revalidate: 3600` на фронтенде делает для fetch — см. разделы 3 и 8 выше, только на уровне самого API, а не только клиента этого API).

### 10.5 Пагинация

Эндпоинт вида `GET /api/trips`, который сейчас отдаёт _все_ строки таблицы одним ответом, линейно растёт по времени ответа и объёму передаваемых данных вместе с ростом таблицы — при 10 рейсах разницы не видно, при 50 000 это уже медленный запрос и тяжёлый JSON. Пагинация — отдавать данные порциями (`GET /api/trips?page=2&pageSize=20`) с фиксированным (и предсказуемым по производительности) размером ответа независимо от того, сколько всего строк в таблице.

### 10.6 Пулинг соединений (connection pooling)

Установка нового TCP-соединения с Postgres и аутентификация на нём — не бесплатная операция (десятки миллисекунд). Пул соединений держит уже открытые, аутентифицированные соединения переиспользуемыми между запросами вместо того, чтобы открывать новое на каждый HTTP-запрос и закрывать его в конце. Npgsql (провайдер EF Core для Postgres, который мы используем) включает пулинг по умолчанию через саму строку подключения — вопрос не «включить», а «явно ли настроены разумные пределы» (`Maximum Pool Size` и т.п.) под ожидаемую нагрузку, а не оставлены дефолтные значения без проверки.

## 11. Архитектурные шаблоны и паттерны фронтенда

Свод того, что реально используется в `busbooking-frontend/` на сегодня (не теория — с привязкой к конкретным файлам).

### 11.1 App Router: Server/Client Components вместо единого рендер-режима

Каждый файл `page.tsx`/компонент сам решает, где он выполняется — нет глобального переключателя «весь сайт SPA» или «весь сайт SSR» (см. раздел 3). По умолчанию (без `"use client"`) — серверный компонент: выполняется на сервере, может быть `async function` и напрямую делать `await fetch(...)`, ничего не отправляет в браузер как JS-код (`app/page.tsx`, `app/routes/page.tsx` — `Home`/`RoutesPage` асинхронные, дёргают `getRoutes()`/`getRoutesVM()` прямо в теле функции). Директива `"use client"` в начале файла (`Header.tsx`, `SearchWidget.tsx`, `login/page.tsx`) — единственный способ получить интерактивность (`useState`, обработчики событий, браузерные API); эти компоненты гидрируются в браузере и увеличивают JS-бандл.

### 11.2 Паттерн `index + vm` для интерактивных страниц

Страницы с нетривиальной клиентской логикой разбиты на пару файлов: `page.tsx` (только JSX — разметка и биндинг к значениям из хука) + `use<Название>PageVM.ts` (весь стейт, обработчики, побочные эффекты, вызовы API). Пример — `app/login/page.tsx` + `app/login/useLoginPageVM.ts`, `app/signup/` — та же пара. VM-хук возвращает плоский объект (`{ email, setEmail, password, setPassword, error, isSubmitting, handleSubmit }`), который `page.tsx` целиком деструктурирует — сама разметка не содержит ни одного `useState`/`fetch` напрямую. Это тот же принцип, что и `pages/CategoriesPage` + `useCategoriesPageVM` из референсного Vite-проекта (раздел 1) — просто адаптированный под файловую структуру `app/`.

Не путать с `routes.vm.ts` в `app/routes/` — там `getRoutesVM` не React-хук, а обычная асинхронная функция для _серверного_ компонента (фильтрация списка маршрутов по `searchParams`), имя `vm` то же самое по духу (view-model — данные, подготовленные специально под конкретную страницу), но без стейта и хуков React, потому что серверный компонент не может их использовать.

### 11.3 Тонкий API-клиент на ресурс: один файл — один backend-контроллер

`api/*.ts` — не один универсальный HTTP-клиент, а по файлу на каждый ресурс бэкенда, зеркалящему конкретный контроллер: `api/routes.ts` ↔ `RoutesController`, `api/trips.ts` ↔ `TripsController`, `api/seats.ts` ↔ `SeatsController`, `api/payments.ts` ↔ `PaymentsController`, `api/bookings.ts` ↔ `BookingsController`, `api/auth.ts` ↔ `AuthController`. Каждый файл экспортирует чистые функции (`getRoutes()`, `searchTrips(params)`, `bookSeat(payload)`), без классов/состояния — вызывающему компоненту всё равно, что происходит внутри (`fetch`, кэш-заголовки, авторизация), он получает уже типизированный результат или брошенный `ApiError`.

Все они построены поверх двух общих слоёв (`api/client.ts`):

- **`apiFetch<T>`** — база: собирает URL из `NEXT_PUBLIC_API_BASE_URL`, кидает `ApiError` (с `status`) на не-2xx, разбирает JSON/`204`. Используется напрямую для _публичных_ эндпоинтов (`routes.ts`, `trips.ts` — не требуют токена).
- **`authFetch<T>`** (`api/authFetch.ts`) — декоратор поверх `apiFetch`: подставляет `Authorization: Bearer <accessToken>` из `auth/tokenStorage.ts` и делает `window.location.href = "/login?redirectTo=..."` при отсутствии токена или `401` от сервера. Используется для JWT-защищённых эндпоинтов (`seats.ts`, `payments.ts`, `bookings.ts`). Явно клиентский модуль (читает `localStorage`, `window.location`) — не для серверных компонентов.

### 11.4 Типизированные ошибки вместо строк

`ApiError extends Error` с полем `status` (`api/client.ts`) — введено, когда понадобилось различать `409 Conflict` (место перехвачено другим пользователем — можно восстановиться, перезапросить список мест) от прочих ошибок (`500`, сеть недоступна — просто показать сообщение). До этого код на любую ошибку API просто ловил `catch { setError("что-то пошло не так") }`, не заглядывая внутрь — этого достаточно для форм логина/регистрации, но недостаточно для флоу бронирования, где разные статусы требуют разной реакции UI.

### 11.5 Типы — зеркало backend VM, один файл на сущность

`types/*.ts` (`route.ts`, `trip.ts`, `seat.ts`, `order.ts`, `auth.ts`) один-в-один повторяют форму ответа конкретных `*VM.cs` с бэкенда (`RouteSummaryVM` → `RouteSummary`, `TripCreateVM` → `TripSearchResult`, `SeatDetailVM` → `Seat`, и т.д.) — поля называются так же (в camelCase — так ASP.NET Core сериализует JSON по умолчанию), просто без runtime-валидации (TypeScript типы стираются при компиляции, это контракт для разработчика/тайпчекера, не проверка данных в рантайме). Общие маленькие типы, переиспользуемые в нескольких местах (`Weekday` — см. раздел про `types/route.ts` в истории правок), выносятся в тот файл, где впервые появились, и импортируются оттуда, а не дублируются.

### 11.6 Context + Provider для кросс-компонентного состояния авторизации

`auth/AuthProvider.tsx` (React Context) — единственное на весь фронтенд глобальное клиентское состояние. Не Redux/Zustand/глобальный store общего назначения (осознанно, см. раздел 1 — «стора нам не понадобится вовсе на старте»), а узкоспециализированный контекст ровно под одну вещь: текущий пользователь + `signIn`/`signUp`/`signOut`. Любой компонент читает его через `useAuth()` (`auth/useAuth.ts`) — тонкая обёртка над `useContext`, которая кидает осмысленную ошибку, если компонент случайно отрендерен вне `<AuthProvider>`, вместо непонятного `null.user`.

## 12. Общий каталог архитектурных паттернов и применимость к BusBooking

Разбор стандартного списка архитектурных паттернов (монолитные/структурные, распределённых систем, UI/разделения ответственности, отказоустойчивые/cloud-native) — что из этого у нас уже есть (пусть и в лёгкой форме), а что осознанно не нужно на MVP-масштабе и почему. Проект: веб-сервис бронирования автобусных билетов, один бэкенд (ASP.NET Core API), один фронтенд (Next.js), одна БД (Postgres), одна внешняя интеграция (Stripe) — не распределённая система, команда небольшая, приоритет — скорость разработки MVP, а не готовность к нагрузке уровня «десятки тысяч RPS».

### 12.1 Монолитные и структурные

**Многослойная (Layered) архитектура** — приложение делится на горизонтальные уровни, каждый общается только с соседним.
**У нас: уже есть, по факту.** Бэкенд: `Controller → Repository/UnitOfWork → EF Core → Postgres` (раздел 10 выше), плюс `ServiceManager` как отдельный слой для вспомогательных сервисов (токены, пароли, платежи). Фронтенд: `page.tsx (View) → useXxxPageVM (логика) → api/*.ts (доступ к данным)` — та же идея, просто три уровня вместо классических «презентация/бизнес-логика/данные». Ничего добавлять не нужно, уже структура проекта.

**Клиент-сервер (Client-Server)** — независимые клиенты запрашивают ресурсы у сервера.
**У нас: тривиально есть по построению.** Next.js фронтенд (клиент, хоть и с серверным рендерингом) обращается к отдельному ASP.NET Core API (серверу) по HTTP/JWT — ровно эта модель, без вариантов, раз выбраны отдельные фронтенд и бэкенд (см. `ROADMAP.md`, обоснование JWT именно из-за разных доменов).

**Каналы и фильтры (Pipes and Filters)** — данные проходят цепочку независимых компонентов-фильтров.
**У нас: не нужен как отдельный архитектурный паттерн**, но микроскопическая версия идеи уже есть в ASP.NET Core middleware pipeline (`app.UseAuthentication() → app.UseAuthorization() → app.UseRateLimiter() → ...` в `Program.cs`) — это и есть каналы-и-фильтры, просто предоставлено фреймворком, не наш код. Полноценный вариант (конвейер трансформации бизнес-данных, не HTTP-запросов) не подходит — у нас нет многоэтапной обработки данных такого рода (один заказ не проходит через цепочку независимых трансформаций).

### 12.2 Распределённых систем

**Микросервисная архитектура** — набор слабо связанных, независимо деплоящихся сервисов.
**Не нужна.** Один бэкенд, одна команда, один деплой — микросервисы добавили бы сетевые вызовы между сервисами вместо вызовов метода, отдельную оркестрацию (Kubernetes/service mesh), распределённые транзакции для того, что сейчас — один `_unitOfWork.Save()`. Оправдано только когда разные части системы должны масштабироваться/деплоиться независимо разными командами — у нас этого нет и не предвидится на горизонте MVP.

**Событийно-ориентированная (Event-Driven)** — система реагирует на асинхронные события через шину сообщений.
**Частично уже есть, в зачаточной форме — вебхук Stripe.** `PaymentsController.HandleWebhook` — это по сути обработчик входящего события (`checkout.session.completed`) от внешней системы, просто через HTTP callback, а не через message broker. Полноценная событийная архитектура (свой event bus, публикация внутренних доменных событий типа `OrderConfirmed` для нескольких независимых подписчиков) стала бы оправдана, когда появятся реально независимые побочные эффекты на одно событие — например, у нас в `ROADMAP.md` уже запланированы «email-уведомление» + «SMS» + «напоминание перед отправлением» на событие подтверждения заказа (Фаза 5) — **вот тогда** имеет смысл присмотреться к паттерну (например, просто `IEventPublisher`-абстракция с несколькими обработчиками внутри одного процесса — не обязательно сразу тянуть RabbitMQ/Kafka).

**Брокер (Broker)** — посредник координирует коммуникацию между разъединёнными компонентами.
**Не нужен.** Требует несколько независимых сервисов, которые нужно координировать — у нас один бэкенд-процесс, посредничать не между кем.

### 12.3 UI и разделения ответственности

**MVC (Model-View-Controller)** — данные, UI и обработчик ввода разделены.
**Исторически был, сейчас не используется намеренно.** Бэкенд начинался как ASP.NET Core MVC (`Views/`, `Controllers`, cookie-аутентификация) — полностью снесён в Фазе 4 `ROADMAP.md` при переходе на чистый JSON API (`API_JWT_MIGRATION.md`). Сейчас контроллеры остались только как HTTP-роутинг+валидация, а весь `View`-слой — ответственность фронтенда (другой процесс, другой репозиторий). Возвращать MVC на бэкенд не нужно — разделение фронтенд/бэкенд уже даёт то же самое разделение ответственности, но по сетевой границе, а не по слоям одного процесса.

**MVVM (Model-View-ViewModel)** — двусторонний биндинг между UI и бизнес-логикой, популярен в клиентских приложениях.
**У нас уже есть, в облегчённой форме — паттерн `index + vm`** (раздел 11.2 выше): `page.tsx` — View (только JSX), `useXxxPageVM.ts` — ViewModel (состояние + обработчики), `types/*.ts` — Model (форма данных). Не классический MVVM с автоматическим двусторонним биндингом (React вообще так не работает — однонаправленный поток данных, `value`+`onChange` вместо биндинга), но роль та же: View ничего не знает про то, как получены данные, ViewModel ничего не знает про разметку.

**CQRS (Command Query Responsibility Segregation)** — раздельные модели для команд (запись) и запросов (чтение).
**Не нужен на этом масштабе.** У нас и так почти CQRS-подобно на уровне контроллеров (`GetXxx`-эндпоинты только читают через `GetAll`/`GetById`, `PostXxx`/`PutXxx` только пишут) — но это естественное следствие REST, а не осознанный CQRS. Полноценный CQRS (отдельная модель чтения, denormalized read store, event-sourced запись) оправдан при сильно разной нагрузке чтения/записи или сложных read-моделях, агрегирующих данные из многих источников — у нас один `Order`/`Trip`/`SeatDetail` в одной таблице каждый, читаются и пишутся напрямую через EF Core, разделять не от чего выигрывать.

### 12.4 Отказоустойчивые / cloud-native

**Circuit Breaker** — временно блокирует вызовы отказавшего/перегруженного сервиса, чтобы не множить сбои.
**Стоит добавить в будущем, не сейчас** — единственная точка, где это реально применимо: вызовы Stripe API из `PaymentService` (единственная внешняя сетевая зависимость в проекте). Сейчас при недоступности Stripe каждый запрос на `/api/payments/checkout/{orderId}` будет заново ждать таймаут вместо того, чтобы быстро отказать после N подряд неудач. Библиотека `Polly` (`.NET`, интегрируется как `IHttpClientFactory`-политика) — стандартный выбор для этого в ASP.NET Core, когда/если Stripe станет заметной точкой отказов в проде. Зафиксировать как пункт в `ROADMAP.md` (Фаза 10.6 — «отложить до роста нагрузки») имеет смысл, делать сейчас — нет, при текущем трафике риск не оправдывает сложность.

**Event Sourcing** — состояние приложения хранится как последовательность событий, а не как текущий снимок; текущее состояние — производное (replay событий).
**Не нужен, существенно overkill.** Требует полностью другой модели хранения данных (append-only event store вместо мутируемых таблиц), сильно усложняет запросы (текущее состояние `Order` нужно каждый раз проецировать из истории событий, а не просто читать строку) — оправдан, когда критична полная история изменений с возможностью replay/аудита на любой момент времени (финтех, банкинг). У нас есть похожая, но кардинально более простая потребность — «аудит-лог админ-действий» (`ROADMAP.md`, Фаза 2, ещё не сделан) — для неё достаточно обычной append-only таблицы `AuditLog`, Event Sourcing для этого не требуется.

### Итог

Ничего добавлять прямо сейчас не нужно — то, что применимо к нашему масштабу (Layered, Client-Server, MVVM-подобный `index+vm`), уже реализовано органически, без того чтобы кто-то осознанно «выбирал паттерн». Единственный пункт, который стоит держать в поле зрения на будущее — **Circuit Breaker вокруг Stripe-вызовов**, когда платёжный трафик вырастет настолько, что временная недоступность Stripe станет заметной проблемой, а не гипотетической.

## 13. Что такое DDD и статья про архитектурные паттерны (systems.education)

Источник: [systems.education/wis_ddd_architectural_patterns](https://systems.education/wis_ddd_architectural_patterns) — это конкретно глава 6 книги Владимира Хононова _«What is Domain-Driven Design?»_ (O'Reilly), не весь DDD целиком. Сама глава разбирает **три архитектурных паттерна и то, какой стиль реализации бизнес-логики каждый из них предполагает** — а не объясняет DDD с нуля (это делают более ранние главы той же книги). Поэтому ниже — сначала общее объяснение DDD (не из этой статьи, общеизвестная методология), потом разбор конкретно того, что описывает статья.

### 13.1 Что такое DDD (Domain-Driven Design)

DDD — предметно-ориентированное проектирование, методология из книги Эрика Эванса _«Domain-Driven Design»_ (2003): подход к проектированию софта, где **структура кода строится вокруг бизнес-домена** (реальных правил и процессов предметной области — у нас это было бы «бронирование мест», «удержание места на 15 минут», «маршрут с расписанием»), а не вокруг технических слоёв (контроллеры/репозитории/таблицы) или структуры базы данных. Ключевые идеи:

- **Ubiquitous language (единый язык)** — код называет вещи так же, как их называют бизнес-эксперты в разговоре, без перевода «на язык программистов». Если бизнес говорит «удержание места», в коде должен быть класс/метод `SeatHold`, а не абстрактный `SeatDetail.IsReserved = true`.
- **Bounded context (ограниченный контекст)** — большая система делится на границы, внутри которых одно и то же слово значит одно и то же (в разных контекстах «заказ» может означать разные вещи — для бронирования это «место+рейс+статус оплаты», для бухгалтерии — «финансовая транзакция»). Внутри одной границы — своя модель, свой язык, не обязательно совпадающий с соседними.
- **Entity** — объект с идентичностью, которая сохраняется при изменении атрибутов (`Order` с конкретным `Id` — тот же заказ, даже когда `Status` меняется с `PendingPayment` на `Confirmed`).
- **Value Object** — объект без собственной идентичности, сравнивается по значению (например, деньги/диапазон дат — два объекта с одинаковыми полями взаимозаменяемы).
- **Aggregate** — кластер связанных Entity/Value Object с одним «корнем» (aggregate root), через который идут все изменения, чтобы гарантировать целостность правил (например, `Order` как корень, а `OrderSeat` — часть агрегата, не изменяется в обход `Order`).
- **Domain events** — факт, что что-то значимое для бизнеса произошло (`OrderConfirmed`, `SeatHoldExpired`) — на них могут реагировать другие части системы (см. раздел 12.2 выше про событийную архитектруру).
- **Repository** — абстракция доступа к агрегатам, скрывающая, что они на самом деле хранятся в БД (у нас это буквально `IRepository<T>`/`IUnitOfWork` — то же название, тот же смысл).

DDD — не архитектурный паттерн сам по себе, а способ **проектировать модель предметной области**; архитектурные паттерны (главы этой книги) — про то, **как эту модель разместить относительно инфраструктуры** (БД, UI, внешние сервисы).

### 13.2 Три паттерна из статьи

**Layered Architecture (многослойная архитектура)** — код делится на технические слои: presentation (UI) → business logic → data access, зависимости идут строго вниз (business logic зависит от data access). По статье — подходит для систем, использующих **Active Record**: бизнес-сущности сами знают, как себя сохранять (методы вроде `order.Save()` прямо на самой сущности, слитой с ORM).

**Ports and Adapters (Hexagonal Architecture)** — переворачивает Layered: бизнес-логика в центре, ничего не знает про инфраструктуру. **Ports** — интерфейсы, которые объявляет сама бизнес-логика (например, «мне нужен способ сохранить заказ», не «мне нужен EF Core»); **adapters** — конкретные реализации под конкретную технологию (Postgres-адаптер, in-memory-адаптер для тестов). По статье — подходит для систем, реализующих **Domain Model pattern**: богатая доменная модель с собственными правилами, полностью изолированная от того, как и где данные физически хранятся.

**CQRS (Command Query Responsibility Segregation)** — одни и те же данные представлены **несколькими персистентными моделями**: одна модель команд (source of truth, строго консистентная, обрабатывает изменения состояния) и одна или несколько read-моделей/проекций (только для чтения, оптимизированы под конкретные запросы). Синхронизация проекций — либо синхронно (проекция сама опрашивает модель команд по чекпоинтам), либо асинхронно (модель команд публикует события в шину сообщений, проекции подписаны на них — сложнее, но масштабируется лучше).

### 13.3 Применимость к BusBooking

**Сейчас у нас — Active Record-style Layered, не Domain Model.** `Repository<T>`/`UnitOfWork` оборачивают EF Core сущности почти без бизнес-логики внутри самих сущностей (`Order`, `SeatDetail` — по сути анемичные модели данных, вся логика в контроллерах: `SeatsController.BookSeat`, `PaymentsController.ConfirmOrderForSession`). Это ровно тот случай, для которого статья рекомендует именно **Layered**, а не Hexagonal — и у нас он уже используется (раздел 12.1 выше), осознанно ничего менять не нужно.

**Ports and Adapters (Hexagonal) — не оправдан сейчас.** Стал бы уместен, если бы бизнес-правила бронирования (seat hold на 15 минут, переходы `OrderStatus`, инварианты вроде «нельзя подтвердить заказ без оплаты») выросли настолько сложными, что их стоило бы вынести в отдельную доменную модель, независимую от EF Core — тестируемую без БД вообще, с адаптером Postgres как деталью, которую можно подменить. При текущем размере логики (несколько `if`-проверок в контроллере) это усложнение не окупается — тот же вывод, что и в разделе 12.

**CQRS — не оправдан**, разбор уже есть в разделе 12.3 (то же самое, статья не добавляет новых аргументов «за» для нашего масштаба, только подтверждает, что CQRS — это буквально «несколько персистентных моделей одних и тех же данных», а не просто «раздельные C#-классы для чтения/записи», что делает его тяжелее внедрить, чем кажется по названию).

- **Circuit Breaker** (библиотека `Polly`) вокруг вызовов Stripe API в `PaymentService` — уже разобрано в разделе 12.4, реальный кандидат, когда вырастет платёжный трафик.
- **Strategy явно** (не неявно, как в 14.1) — для мультивалютности (`ROADMAP.md`, Фаза 5): разные стратегии конвертации/источники курсов валют за одним интерфейсом `ICurrencyConversionStrategy`, выбираемым в рантайме.
- **Observer / Domain Events** — на подтверждение заказа: сейчас `PaymentsController.ConfirmOrderForSession` делает всё последовательно в одном методе; когда появятся email/SMS/напоминание (`ROADMAP.md`, Фаза 5), лучше публиковать одно событие `OrderConfirmed` с несколькими независимыми подписчиками, а не наращивать один метод новыми вызовами.
- **Options pattern** (`IOptions<StripeSettings>`/`IOptions<JwtSettings>` вместо `configuration["Stripe:SuccessUrl"]` строками россыпью по `PaymentService`/`TokenService`) — идиоматичная для ASP.NET Core версия того, что уже делается вручную; не новый паттерн по сути, а типобезопасная замена текущего подхода.
- **HOC / `withAuthGuard`** на фронтенде — уже запланирован, но не реализован (`FRONTEND_AUTH_PLAN.md`): обёртка над клиентскими компонентами, которая сама делает редирект на `/login`, вместо того чтобы каждая JWT-защищённая страница писала эту проверку внутри себя вручную (сейчас именно так — см. `/trips/[tripId]/seats`).
- **Specification pattern** (опционально, низкий приоритет) — сейчас фильтры репозиториев передаются как сырые `Expression<Func<T, bool>>` лямбды прямо из контроллеров (`TripsController.SearchTrips`); можно вынести повторяющиеся условия в именованные объекты-спецификации, но при текущем количестве фильтров выигрыш небольшой — не срочно.

## 15. Что такое raw SQL в миграциях EF Core

Возникло при схлопывании миграций backend-проекта в одну (`Migrations/InitialCreate.cs`) — часть индексов пришлось переносить руками, потому что автогенерация их не воспроизвела. Дальше — что это значит.

### 15.1 Обычный режим EF Core: миграция генерируется из модели

По умолчанию Entity Framework Core работает так: вы описываете сущности на C# (`Entity/User.cs`, `Fluent API` в `ApplicationDbContext.OnModelCreating`), а когда меняете модель и запускаете `dotnet ef migrations add <Имя>`, EF **сам** сравнивает новое описание модели со старым снепшотом (`ApplicationDbContextModelSnapshot.cs`) и генерирует C#-код миграции — `migrationBuilder.CreateTable(...)`, `AddColumn(...)`, `CreateIndex(...)` и т.д. Это не SQL напрямую, а C#-DSL (domain-specific язык), который EF потом сам транслирует в SQL нужного диалекта (Postgres в нашем случае) при выполнении.

Пример из нашего же `InitialCreate.cs` — обычный, «декларативный» способ задать индекс:

```csharp
migrationBuilder.CreateIndex(
    name: "IX_Trips_RouteId",
    table: "Trips",
    column: "RouteId");
```

EF знает, как это выполнить на любой поддерживаемой БД — сам построит правильный `CREATE INDEX` под конкретный диалект SQL.

### 15.2 Raw SQL — когда декларативного способа нет

**Raw SQL** — это когда вместо C#-DSL миграция содержит **буквальный текст SQL-запроса**, который EF просто выполняет как есть, ничего не интерпретируя и не транслируя:

```csharp
migrationBuilder.Sql(
    "CREATE INDEX \"IX_User_Email_Lower\" ON \"User\" (lower(\"Email\"));");
```

Это нужно, когда то, что требуется сделать, **невозможно выразить** через модель сущностей и Fluent API — у EF Core просто нет C#-метода для этой конкретной вещи. Наш случай: `AuthController.SignIn`/`SignUp` сравнивают email через `.ToLower()`, что транслируется в SQL как `lower("Email") = lower(@p)`. Обычный индекс на колонку `Email` для такого сравнения бесполезен — планировщик Postgres не сможет им воспользоваться, потому что физически в индексе лежат значения `Email` как есть, а не их lowercase-версия. Нужен **функциональный (expression) индекс** — индекс не по значению колонки, а по результату выражения над ней (`lower("Email")`). Fluent API/`HasIndex()` умеет строить индексы только по колонкам напрямую, выражений он не поддерживает — поэтому единственный способ создать такой индекс через EF Core Migrations — написать `CREATE INDEX ... (lower(...))` как чистый SQL-текст.

### 15.3 Практическое следствие: raw SQL не входит в модель EF

Из-за того что `migrationBuilder.Sql(...)` — это просто «выполнить эту строку SQL», EF **не знает**, что этот индекс существует — он не отражён в C#-модели (`ApplicationDbContextModelSnapshot.cs`), только физически в БД. Отсюда конкретный урок из нашей же сессии: когда мы схлопывали 6 файлов миграций в один свежий `InitialCreate` (командой `dotnet ef migrations add InitialCreate` после удаления старых миграций), EF пересобрал новую миграцию **строго из текущей C#-модели** — и оба expression-индекса (`lower(Email)`, `lower(Username)`) **не попали** в неё автоматически, потому что для EF их как бы не существует. Пришлось вручную скопировать `migrationBuilder.Sql(...)`-строки из старой миграции (`AddAuthLookupIndexes`, которой уже не стало) в новую. Общее правило: **любой raw SQL в миграциях нужно переносить руками** при любой операции, которая регенерирует миграции из модели (squash, `migrations remove` + пересоздание и т.п.) — авто-инструменты о нём просто не знают.

## 16. Как работает Stripe Checkout Session — весь workflow

Полный путь заказа от брони места до подтверждённой оплаты — какие шаги происходят на нашем бэкенде, какие на стороне Stripe, и почему это два независимых канала.

### 16.1 Бронирование места (до всякой оплаты)

`POST /api/seats/book` (`SeatsController`) резервирует место на 15 минут (`SeatDetail.IsReserved`/`ReservedUntil`) и создаёт `Order` со статусом `PendingPayment`. На этом шаге про Stripe ещё ничего не известно — это чисто наша внутренняя бронь.

### 16.2 Фронтенд запрашивает checkout-сессию

`POST /api/payments/checkout/{orderId}` (`PaymentsController.CreateCheckoutSession`):

- Проверяет, что заказ принадлежит текущему пользователю (`order.UserId == userId`) и что его вообще ещё можно оплатить (`Status == PendingPayment`, иначе `409`).
- Вызывает `PaymentService.CreateCheckoutSession(order)` — создаёт **Stripe Checkout Session** через Stripe API. Это отдельная, размещённая на серверах Stripe страница оплаты (`SessionCreateOptions`: сумма из `order.TotalPrice`, `SuccessUrl`/`CancelUrl` с `?orderId=...`, `ClientReferenceId = order.Id`).
- Сохраняет `session.Id` в `order.StripeSessionId` — ключ, по которому webhook потом найдёт нужный заказ.
- Возвращает клиенту `{ checkoutUrl }` — URL страницы оплаты на `checkout.stripe.com`.
- Если Stripe недоступен (`StripeException`, например неверный/отсутствующий `SecretKey` в деве) — отвечает `502` с понятным сообщением, а не сырой стектрейс.

### 16.3 Редирект браузера на Stripe

`/checkout/[orderId]` (`useCheckoutPageVM.ts`) вызывает этот эндпоинт и делает `window.location.href = checkoutUrl` — настоящий переход на внешний домен, не `router.push` (это не наш маршрут, Next.js тут ни при чём).

### 16.4 Оплата — целиком на стороне Stripe

Форма ввода карты и сам платёж происходят на инфраструктуре Stripe. Наш бэкенд в этот момент ничего не знает и не может знать о результате напрямую — у него просто нет канала связи с формой оплаты, только со Stripe API/вебхуками.

### 16.5 Подтверждение — асинхронный вебхук от Stripe

После успешной оплаты Stripe сам делает `POST /api/payments/webhook` (эндпоинт публичный, `[AllowAnonymous]` — его вызывает не браузер пользователя, а серверы Stripe):

- `ConstructWebhookEvent` проверяет подпись запроса (заголовок `Stripe-Signature` против секрета `Stripe:WebhookSecret`) — без этой проверки кто угодно мог бы подделать POST-запрос с фейковым «оплата прошла».
- На событие `checkout.session.completed` вызывается `ConfirmOrderForSession`: находит `Order` по `StripeSessionId`, переводит в `Confirmed`, сохраняет `PaymentIntentId` (пригодится для возврата при отмене, см. `CANCELLATION_REFUND_PLAN.md`), и подтверждает места (`SeatDetail.ConfirmSeats` — снимает временный hold, ставит `IsOccupied = true` уже навсегда).
- Обработка идемпотентна: если заказ уже не в статусе `PendingPayment` (Stripe добросовестно повторно доставляет один и тот же вебхук, это стандартная практика, не баг) — метод просто ничего не делает при повторном вызове.

### 16.6 Success/cancel редирект — отдельный, независимый канал

Одновременно (но не синхронно) Stripe редиректит браузер пользователя на `SuccessUrl`/`CancelUrl` — у нас это `http://localhost:3000/booking/success`/`/booking/cancel`, жёстко заданные в `appsettings.json`. Наши страницы `app/booking/success`/`app/booking/cancel` просто показывают сообщение по `?orderId=` из query-строки — реального похода в API они не делают (осознанное решение, см. `BOOKING_FLOW_PLAN.md`, раздел 5).

### 16.7 Ключевая идея: источник истины — вебхук, не редирект

Вебхук (16.5) и success-редирект (16.6) — два **разных, независимых** канала: один сервер→сервер (Stripe напрямую бьёт в наш `/api/payments/webhook`), другой браузер пользователя (Stripe редиректит вкладку). Пользователь теоретически может закрыть вкладку/потерять связь до того, как редирект долетит — тогда success-страница вообще не откроется, но заказ всё равно будет подтверждён, потому что вебхук доставляется независимо от браузера (и Stripe его ретраит при сбоях доставки). Именно поэтому подтверждение заказа (`ConfirmOrderForSession`) происходит по вебхуку, а не по факту показа success-страницы — редирект существует только для UX (что показать пользователю), а не как триггер бизнес-логики.

## 17. `"use client"` — что это значит на самом деле

Кратко это уже упоминалось в разделе 11.1 — здесь разбор подробнее, на конкретном свежем примере (`components/routes/RouteMap.tsx`/`LeafletRouteMap.tsx`).

### 17.1 По умолчанию файл в `app/` — серверный компонент

Без директивы `"use client"` в начале файла компонент выполняется **на сервере** (при сборке для статических страниц, или на каждый запрос для динамических) и превращается в обычный HTML/JSON до того, как долетает до браузера:

- может быть `async function` и напрямую делать `await fetch(...)`/обращаться к БД — без `useEffect`, без состояния загрузки на клиенте (`app/page.tsx`, `app/routes/page.tsx`).
- **не может** использовать `useState`, `useEffect`, `onClick` и вообще любые браузерные API (`window`, `document`, `localStorage`) — на сервере их просто не существует.
- его код (и импортированные им npm-пакеты) **не попадает** в JS-бандл, который скачивает браузер — сервер уже всё посчитал, клиенту нужен только результат (HTML).

### 17.2 `"use client"` — граница, а не место, а маркер целого поддерева

Директива в первой строке файла помечает этот модуль (и всё, что он экспортирует) как код, который должен выполняться в браузере — там, где есть `useState`, обработчики событий (`onClick`, `onSubmit`), `useEffect`, доступ к `window`/`localStorage`. React физически гидрирует такой компонент в браузере: сервер по-прежнему рендерит его в HTML для первой отрисовки (если он не обёрнут в `dynamic(..., { ssr: false })`, см. 17.3), но после загрузки страницы браузер скачивает его JS-код и «оживляет» интерактивность поверх готового HTML.

**Важно понимать про границу:** `"use client"` не изолирует один компонент — он помечает точку входа в клиентское поддерево. Всё, что этот клиентский компонент импортирует и рендерит дальше (даже если в дочернем файле нет своей директивы `"use client"`), тоже становится частью клиентского бандла. Обратное направление работает без ограничений: серверный компонент может передавать клиентскому компоненту пропсы (сериализуемые данные — строки/числа/объекты, не функции и не классы), в точности как `RoutesPage` (серверный, `async`) передаёт уже загруженный `routes: RouteSummary[]` клиентскому `<RouteMap routes={routes} />`.

### 17.3 Когда `"use client"` одной директивы недостаточно — библиотеки, завязанные на `window`

Практический пример из этой сессии — `Leaflet` (карта маршрутов, Фаза 7 `ROADMAP.md`). Библиотека обращается к `window`/`document` **на этапе импорта модуля**, не только при рендере — а во время SSR (даже внутри клиентского компонента при первом серверном проходе для генерации HTML) `window` не существует, и импорт упал бы с ошибкой. Поэтому одного `"use client"` мало — нужно ещё и **полностью исключить компонент из серверного рендеринга**:

```tsx
// components/routes/RouteMap.tsx
"use client";
import dynamic from "next/dynamic";

const LeafletRouteMap = dynamic(
  () => import("./LeafletRouteMap").then((mod) => mod.LeafletRouteMap),
  { ssr: false, loading: () => <Skeleton /> },
);
```

`dynamic(..., { ssr: false })` в App Router **разрешён только внутри уже клиентского компонента** — если написать это прямо в серверном `RoutesResultsSection` (`app/routes/page.tsx`), Next.js откажется собирать проект (ssr: false бессмысленен там, где и так не рендерится на клиенте — по определению серверный компонент не гидрируется). Поэтому структура вышла в два файла:

- `RouteMap.tsx` — тонкий клиентский компонент, единственная задача которого — динамически (без SSR) подгрузить настоящую карту и показать скелетон, пока она грузится.
- `LeafletRouteMap.tsx` — сам `MapContainer`/`TileLayer`/`Marker` из `react-leaflet`, гарантированно рендерится только в браузере.

### 17.4 Практическое правило

Добавлять `"use client"` тогда и только тогда, когда компоненту реально нужны `useState`/`useEffect`/обработчики событий/браузерные API — не «на всякий случай» и не потому, что так проще: каждый такой файл (и всё, что он тянет за собой) увеличивает JS-бандл, который скачивает и выполняет браузер. Публичные страницы (`Home`/`About`/`Routes`) в этом проекте осознанно остаются серверными компонентами целиком, кроме маленьких интерактивных островков внутри (`SearchWidget`, `Header`, теперь `RouteMap`) — сама страница как была статическим/ISR-HTML, так и осталась.

## 18. Как запустить проект локально

Два процесса — backend (ASP.NET Core + Postgres) и frontend (Next.js), плюс опционально Docker.

### 18.1 Backend + Postgres

Вариант А — весь backend в Docker (`docker-compose.yml` в корне репозитория, `BusBooking/`, не `busbooking-frontend/`):

```bash
docker compose up -d          # поднимает Postgres + собранный контейнер API на хосте :5050
```

Вариант Б — Postgres в Docker, backend локально через `dotnet run` (даёт горячую перезагрузку кода, чем и пользовались при разработке в этой сессии):

```bash
docker compose up -d postgres
cd BusBooking
dotnet ef database update                              # применить миграции
dotnet run --urls http://localhost:5100                 # см. ниже про порт
```

**Важно про порты:** `docker-compose.yml` пробрасывает контейнеризованный API на **`:5050`**, а `busbooking-frontend/.env.local` (`NEXT_PUBLIC_API_BASE_URL`) сейчас указывает на **`http://localhost:5100`** — это порт локального `dotnet run` (вариант Б), не докер-контейнера. Если поднимаете backend только через `docker compose up -d` целиком (вариант А, порт 5050) — либо поменяйте `NEXT_PUBLIC_API_BASE_URL` в `.env.local` на `5050`, либо (что удобнее для разработки с горячей перезагрузкой) используйте вариант Б и `dotnet run --urls http://localhost:5100`, ничего в `.env.local` менять не придётся. Секреты дева (JWT-ключ, строка подключения, если не через `docker compose up -d postgres`) — `dotnet user-secrets list` в `BusBooking/` (см. `UserSecretsId` в `.csproj`).

### 18.2 Frontend

```bash
cd busbooking-frontend
npm install
npm run dev                    # http://localhost:3000, Turbopack, горячая перезагрузка
```

`npm run build && npm run start` — production-сборка (то, что реально будет задеплоено на Vercel, см. `ROADMAP.md`, Фаза 8 — деплой пока не настроен, но локальная production-сборка уже проверяется этой командой).

### 18.3 Проверить, что всё поднялось

```bash
curl http://localhost:5100/health   # {"status":"Healthy"} — backend + Postgres живы
curl -I http://localhost:3000       # 200 — frontend отвечает
```

Дальше — реальные данные: сид (маршрут Sofia↔Berlin) применяется миграцией автоматически, но рейсов (`Trip`) из него ещё нет, пока не вызван генератор — авторизоваться Swagger'ом (`/swagger`, дев-сид админ `xyz@mail.com`/`1234`) и дёрнуть `POST /api/trips/generate` с `{ "weeksAhead": 2 }`, тогда `/trips`/`/routes` на фронтенде покажут реальные рейсы, а не пустой список.

## 19. Admin UI — один репозиторий или отдельный?

При добавлении `/admin/*` (`ADMIN_UI_PLAN.md`) встал вопрос: выносить админку в отдельный Next.js-проект/репозиторий или оставить внутри `busbooking-frontend/` как часть той же кодовой базы. Решение — **оставить в том же репозитории**, гейтить через `auth/AdminGuard.tsx` по роли из JWT. Разбор почему.

### 19.1 Что даёт разделение на два репозитория (в теории)

- Независимый деплой — можно выкатывать админку и публичный сайт по разным циклам, не боясь, что правка в одном сломает второе на общем деплое.
- Разный уровень доступа на инфраструктуре — админку можно спрятать за VPN/IP-allowlist/Basic Auth на уровне хостинга, публичный сайт остаётся полностью открытым.
- Меньше JS в бандле публичного сайта — код админки (формы, таблицы) вообще не долетает до обычного посетителя, даже в виде неиспользуемых чанков.
- Разные команды/темпы разработки — если админку и публичный сайт со временем будут вести разные люди, отдельный репозиторий снижает conflict surface.

### 19.2 Почему это не оправдано именно сейчас

- **Масштаб не тот.** Админка — 6 несложных CRUD-страниц (`routes`/`buses`/`schedules`/`trips`/`users`/`bookings`), не отдельный продукт. Разделение репозиториев — операционные накладные расходы (два `package.json`, два деплоя на Vercel, два набора переменных окружения), которые стоит платить только когда сложность/объём кода это оправдывают.
- **Дублирование неизбежно при разделении.** Компонент-система (`components/ui`), API-клиент (`api/client.ts`, `api/authFetch.ts`), типы (`types/*.ts`), `auth/` (весь JWT-стек) — либо копируются в оба репозитория (расхождение гарантировано через пару месяцев), либо выносятся в общий npm-пакет/monorepo (Turborepo/Nx) — а это уже реальный архитектурный шаг, оправданный при таком объёме кода, каким админка пока не является.
- **JWT между отдельными доменами усложняет, не упрощает.** Сейчас `auth/tokenStorage.ts` — `localStorage`, один и тот же origin для публичного сайта и админки (оба под одним доменом, просто разные пути `/admin/*`). Разнеся на два отдельных Vercel-проекта/доменов, пришлось бы решать: либо дублировать логин (пользователь логинится отдельно в каждом приложении — плохой UX для админа, которому иногда нужно и забронировать самому), либо городить SSO/shared-cookie на уровне поддоменов — сложность, которой сейчас попросту нет, потому что `AdminGuard` читает тот же `useAuth()`, что и всё остальное приложение.
- **Изоляция по коду важнее изоляции по деплою на этом этапе.** Настоящая причина заводить отдельный репозиторий — это когда команды физически мешают друг другу (частые конфликты, разный ритм релизов, разный up-time SLA для админки и паблик-сайта). Ничего из этого пока не про этот проект — один разработчик (точнее, один человек + ассистент), единый релизный цикл.

### 19.3 Что уже даёт нужную изоляцию без отдельного репозитория

- **Ролевой гейт, а не просто auth-гейт** (`auth/AdminGuard.tsx`) — `/admin/*` недоступен не только неавторизованным, но и авторизованным не-админам (редирект на `/`, не на `/login` — см. код гварда и комментарий в нём).
- **Файловая изоляция уже есть внутри `app/`** — весь код админки живёт под `app/admin/**`, ничего из него не импортируется публичными страницами (`app/page.tsx`, `app/routes/page.tsx` и т.д.) — Next.js делает route-based code splitting автоматически на уровне файловой системы `app/`, так что JS-бандл `/admin/routes` не долетает до обычного посетителя `/` без необходимости отдельного проекта ради этого.

### 19.4 Когда пересмотреть это решение

- Админка вырастает в полноценный отдельный продукт (не CRUD-таблицы, а сложные дашборды/аналитика/множество ролей) с собственным темпом изменений.
- Появляется требование инфраструктурной изоляции (например, реальный клиент требует, чтобы админка была вообще недоступна из публичного интернета — VPN-only).
- Команда вырастает настолько, что один репозиторий реально становится источником конфликтов между теми, кто правит паблик-сайт, и теми, кто правит админку.

До тех пор — один репозиторий, один деплой, ролевой гейт внутри приложения.

Lazy<T> откладывает создание объекта до первого реального обращения к .Value, а не создаёт его сразу в конструкторе.

В ServiceManager.cs:
private readonly Lazy<ITokenService> \_tokenService = new(() => new TokenService(configuration));
public ITokenService TokenService => \_tokenService.Value;

- new Lazy<ITokenService>(() => new TokenService(configuration)) — сохраняет фабрику (лямбду), но не вызывает её сразу.
- Только при первом обращении к \_tokenService.Value (то есть при первом использовании свойства TokenService) Lazy<T> реально вызывает new TokenService(configuration), кладёт результат в кэш и возвращает.
- Все последующие обращения к .Value отдают уже закэшированный экземпляр — конструктор TokenService больше не вызывается повторно.
- По умолчанию Lazy<T> ещё и потокобезопасен (thread-safe инициализация — если два потока одновременно дёрнут .Value в первый раз, конструктор всё равно выполнится только один раз).

Зачем это здесь: ServiceManager — единственное место, которое явно new-ит все сервисы (TokenService, PasswordHasherService, PaymentService, PasswordBreachChecker) вместо регистрации каждого в DI-контейнере отдельно. Но ServiceManager регистрируется как AddScoped — значит новый экземпляр на каждый HTTP-запрос. Если бы поля не были Lazy, при каждом запросе создавались бы все сервисы сразу, даже если конкретный запрос использует, скажем, только PasswordHasherService. Lazy откладывает создание до реального использования — контроллер, который вызывает только \_services.PasswordHasherService, никогда не создаст TokenService/PaymentService за этот запрос.
