# Фаза 7 (расширение) — единый дизайн для Home/About/Routes: план

Сравнение (см. обсуждение в сессии) показало, что текущая реализация тянет элементы из **двух разных поколений** дизайн-макетов сразу: `design/*.jpg` (более ранняя итерация — лого-конь, шрифт Cuprum, фото-хиро) и присланные HTML-макеты (более поздняя итерация — иконка `call_to_action`, Fira Sans, токены Tailwind, совпадающие с `busbooking-frontend/app/globals.css`/`colors.css` почти один-в-один). Из-за этого разные страницы визуально не согласованы друг с другом (например, `ContactSection` на About использует Home-вариант, а `MissionSection` держит кнопку из JPG-версии, которой нет в актуальном HTML-макете).

Цель этого плана — не «доделать до макета», а **зафиксировать одну версию дизайн-языка** и привести все три страницы (Home/About/Routes) к ней, явно решив каждое расхождение один раз, а не оставляя смесь.

## Принятые решения (источник истины)

1. **HTML-макеты — источник истины, не JPG.** Их цветовые/spacing/шрифтовые токены уже совпадают с `colors.css`/`tailwind.config`, JPG — устаревшая итерация, к ней не возвращаемся.
2. **Идентика:** иконка логотипа — `call_to_action` (уже используется в `Header.tsx`/`Footer.tsx`/обоих HTML-макетах 1 и 2); иконка `pets` из макета "Interactive Route Map" — не принимается, явный разнобой между генерациями макетов, не реальное расхождение с кодом.
3. **Шрифты:** headline — **Fira Sans** (уже подключён через `next/font` в `app/layout.tsx`), не Cuprum. Cuprum нигде не подключён как реальный шрифт в проекте (только в одном макете, третьем по счёту) — вводить третий шрифт ради одной страницы не оправдано, лишний вес и рассинхрон с остальным сайтом.
4. **Реальных фото нет.** `public/images/` содержит только `bus1.png`/`bus2.png` — ни фото-хиро (аэросъёмка автопарка, закатная дорога), ни фото для карточек новостей (зима/салон/Sofia skyline) не существуют как ассеты. **Решение: везде, где макет ожидает фото — используется существующий градиентный/иконочный треатмент**, не мокап-заглушки с фейковыми `data-alt`-описаниями. Замена на реальные фото — отдельная задача (нужны лицензированные/заказанные фото), не часть этого плана.
5. **Одна версия `ContactSection` на весь сайт** — вариант с Home-макета (полноширинный градиент `from-primary via-primary-container to-deep-navy` + **белая** форма). Тёмная/полупрозрачная версия из About-макета (сплошной `bg-[#46088a]`, инпуты `white/10`) — не принимается, чтобы не было двух разных стилей контактной формы на соседних страницах.
6. **Данные не фабрикуются.** Ярлыки маршрутов вида "Direct • 14h 30m" / "Express • 8h 15m" из макета карты — decorative-фикция, в модели данных нет понятия "тип рейса"/расчётной длительности показа. `ActiveRoutesSidebar` продолжает показывать реальные данные расписания (`formatScheduleSummary`) вместо выдуманных ярлыков.
7. **Второй "Top Contact Bar" над хедером** (из макета "Interactive Route Map") — не принимается, в проекте один `Header.tsx` на всех страницах, вводить его только для одной секции — рассинхрон.
8. **Переключатель валюты/языка (`MDL/EN`, `EN/USD`)** — уже отдельные пункты в `ROADMAP.md` (Фаза 7: мультивалютность/i18n), не решается в рамках этого плана — не дублировать задачу.

## 1. Hero-секции — унифицировать `HeroSection.tsx` и `PageHero.tsx`

Сейчас это два почти идентичных компонента (один для Home, один для About/будущих внутренних страниц) с одинаковым градиентным фоном и radial-highlight — уже согласованы между собой достаточно хорошо, но:

- [ ] `PageHero.tsx` (About, и в будущем — любая внутренняя страница) — заголовок сейчас `text-4xl sm:text-6xl`, в макете About акцент на заголовок гораздо крупнее и весомее. Увеличить до `text-5xl sm:text-7xl` (без ухода в 96px/Cuprum из макета — см. решение 3) — визуально ближе к макету, остаётся в рамках Fira Sans.
- [ ] Рассмотреть, стоит ли `HeroSection.tsx` (Home) и `PageHero.tsx` (About) свести в один компонент с параметром (`fullBleed`/`compact` варианты высоты) — сейчас дублирование почти 1:1 (тот же градиент, тот же radial-highlight `style`). Не обязательно, но уменьшит дублирование кода при следующей правке цветов/эффекта.
- [ ] **Не добавлять** фоновые фото (см. решение 4) — оставить как есть, только визуальный вес заголовка.

## 2. `MissionSection.tsx` (About) — явно решить судьбу кнопки "Reserve now!"

- [ ] Кнопка "Reserve now!" (ведёт на `/routes`) есть в коде и в JPG-макете, но отсутствует в актуальном HTML-макете About. **Решение: оставить** — это единственный явный CTA на странице About, ведущий в реальный флоу бронирования (`/routes`), пользы больше, чем вреда от расхождения с одним конкретным макетом. Задокументировать это здесь явно, чтобы при следующем ревью не считалось забытым багом рассинхрона.
- [ ] Остальное (текст, значок "Licensed & Certified", фото автобуса `bus1.png`) уже соответствует актуальному макету — не трогать.

## 3. `NewsSection.tsx` — унифицировать использование между Home/About

Сейчас один и тот же компонент используется на обеих страницах с разными пропсами (`background="plain"` на Home, дефолт на About) — это нормально, реальное расхождение с макетом в другом:

- [ ] Макет About показывает **заголовок слева + ссылку "View all updates" справа** (не центрированный заголовок, как сейчас на обеих страницах). Добавить опциональный проп `viewAllHref?: string` — если передан, рендерить заголовок в `flex justify-between` с ссылкой справа (вместо `text-center`); если не передан — оставить текущее центрированное поведение (для Home, где в макете заголовок по центру).
- [ ] На `app/about/page.tsx` передать `viewAllHref="/routes"` (единственная существующая страница со списком/расписанием — "News"-страницы как таковой ещё нет, см. `ROADMAP.md`, "Реальные новости (News)", не блокирует этот пункт).
- [ ] Карточки — оставить иконку-плейсхолдер (см. решение 4), не добавлять фото.

## 4. Routes-страница — добавить недостающую "Route Details" секцию

- [ ] Макет "Interactive Route Map" содержит 3-колоночную секцию под картой: Efficiency / Live Tracking / 24/7 Support (иконка + короткий текст, чисто декоративный контент, без зависимости от бэкенда). Сейчас на `app/routes/page.tsx` такой секции нет вообще — добавить, переиспользуя паттерн `WhyPometco.tsx` (Home) как референс структуры (3 колонки, иконка в цветном кружке + заголовок + текст) вместо копирования нового компонента с нуля.
- [ ] Тексты — не копировать дословно из макета (специфично под "Balkan region"/GPS-трекинг, которых на самом деле нет как фичи) — переформулировать под то, что реально есть: например "Efficiency" → про фильтр по дню недели и ближайшие рейсы (реальная фича `RouteExplorer`), "24/7 Support" → реальный телефон поддержки (уже есть в Header/Footer), убрать "Live Tracking"/GPS-фичу, которой в проекте нет вообще (не фабриковать несуществующий функционал, см. решение 6).

## 5. `ActiveRoutesSidebar.tsx` — без изменений по существу

- [ ] Уже показывает реальные данные (`formatScheduleSummary`) вместо выдуманных "Direct/Express" ярлыков — соответствует решению 6 этого плана. Единственное — свериться, что иконки (`local_shipping`/`directions_bus`/`airport_shuttle`, назначаются по индексу `index % ICONS.length`) не выглядят случайными при реальном количестве маршрутов (сейчас 2 реальных + 1 тестовый в БД) — не критично, не менять логику ради 3 маршрутов.

## 6. `ContactSection.tsx` — уже единый вариант, зафиксировать как единственный

- [ ] Явных изменений в коде не требуется — компонент уже переиспользуется одинаково на Home и About (решение 5 этого плана). Убедиться, что при появлении новых страниц с контактной формой (если будут) используется этот же компонент, а не новый вариант.

## 7. Header — свериться, что нет расхождений между страницами

- [ ] `Header.tsx` уже общий для всего сайта (`app/layout.tsx`) — секция "Top Contact Bar" из макета карты не принимается (решение 7). Действий не требуется, пункт — фиксация решения, чтобы при будущей правке Header её случайно не добавили только для одной страницы.

## Порядок выполнения

1. `PageHero.tsx` — увеличить заголовок (п.1).
2. `NewsSection.tsx` — проп `viewAllHref`, применить на `app/about/page.tsx` (п.3).
3. Новая секция "Route Details" на `app/routes/page.tsx`, тексты — под реальный функционал, не под текст макета (п.4).
4. Живая проверка каждой из трёх страниц (Home/About/Routes) в браузере на desktop-ширине — визуально сверить, что теперь нет разностилья между страницами (сравнить скриншоты, не полагаться только на чтение кода — тот же урок, что и с багом карты в `RESPONSIVE_PLAN.md`).
5. Отметить `[+]` здесь и (если применимо) в `ROADMAP.md`.

## Явно не входит в этот план

- Замена градиентных hero/новостных плейсхолдеров на реальные фото — заблокировано на реальные ассеты (см. решение 4), отдельная задача, когда фото появятся.
- Переключатель валюты/языка — уже в `ROADMAP.md`, Фаза 7.
- Объединение `HeroSection.tsx`/`PageHero.tsx` в один компонент — опционально упомянуто в п.1, не обязательно к выполнению, только если удобно по ходу.
