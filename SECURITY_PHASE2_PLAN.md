## 1. Строгий rate limit на `signin`/`signup`

Сейчас в `Program.cs` только один глобальный `GlobalLimiter` (100 запросов/мин на IP на весь API) — этого мало для логина: 100 попыток подбора пароля в минуту вполне реальны.

**Шаги:**
- В `Program.cs` добавить именованную политику через `AddPolicy` (например `"auth"`, sliding/fixed window, 5 запросов/мин на IP) поверх существующего `AddRateLimiter`.
- Навесить `[EnableRateLimiting("auth")]` на `AuthController.SignIn` и `SignUp`.
- Проверить, что `RejectionStatusCode = 429` для этой политики тоже настроен (как у глобальной).
- Проверить вживую: burst-запросами на `/api/auth/signin` (аналогично тому, как уже было протестировано для глобального лимита — 99×200 + 6×429 на границе).

**Сделано.** Политика `"auth"` (5/мин, sliding window) добавлена в `Program.cs`, навешена на `SignIn`/`SignUp` через `[EnableRateLimiting("auth")]`. Проверено вживую: 5×`401` (неверный пароль) + 6-я и 7-я попытки — `429`; несвязанный эндпоинт (`/api/routes`) не задет.

Попутно обнаружен и исправлен блокирующий баг, не связанный с этим пунктом: в незакоммиченной рабочей копии `Program.cs` отсутствовала регистрация `ITokenService`/`IPasswordHasherService` в DI (были в последнем коммите, но убраны в рабочей копии до начала этой сессии) и `IPaymentService` (не был зарегистрирован никогда) — приложение падало на старте. По ходу дела `ServiceManager` переработан так, что сам строит эти три сервиса внутри себя (`new TokenService(configuration)` и т.д.), а не получает их через DI — теперь в `Program.cs` регистрируется только `IServiceManager`, как и просил пользователь.

## 2. Account lockout после N неудачных попыток входа

Не реализовано вообще (в коде нет ни счётчика, ни `LockoutEnd`).

**Шаги:**
- `User`: добавить поля `FailedLoginAttempts` (int), `LockoutEnd` (DateTime?).
- Миграция.
- В `AuthController.SignIn`: перед `VerifyPassword` — если `LockoutEnd` в будущем, вернуть `401`/`423` с понятным сообщением, не тратя время на верификацию пароля.
- При неверном пароле — инкремент счётчика; при достижении порога (например, 5) — выставить `LockoutEnd = now + 15 минут`.
- При успешном входе — сбросить счётчик и `LockoutEnd`.
- Проверить вживую: 5 неверных попыток → 6-я (даже с верным паролем) блокируется до истечения `LockoutEnd`.

**Сделано.** `User.FailedLoginAttempts`/`LockoutEnd` (миграция `AddAccountLockout`, безопасный `ADD COLUMN` с дефолтом). `AuthController.SignIn`: сначала проверка `LockoutEnd` (возвращает `423 Locked` с телом `{ message, lockoutEnd }`, до `VerifyPassword` — не тратим время на bcrypt при уже заблокированном аккаунте), при неверном пароле — инкремент + выставление `LockoutEnd` при достижении `MaxFailedLoginAttempts = 5` (`LockoutDuration = 15 минут`), при успехе — сброс обоих полей.

Проверено вживую полным циклом (пришлось учесть взаимодействие с rate-limit из пункта 1 — 5 попыток укладываются ровно в лимит 5/мин, а 6-й запрос в том же окне сначала ловил бы `429` от лимитера, а не `423` от лок-аута, так что тест разнесён на два перезапуска бэкенда, чтобы сбросить in-memory состояние лимитера, не трогая персистентное состояние блокировки в БД): 5×`401` → `FailedLoginAttempts=5`, `LockoutEnd` выставлен; попытка с **верным** паролем при активной блокировке → `423`, не `200`; после истечения `LockoutEnd` (состарен через SQL) — успешный вход `200`, оба поля сброшены (`0`/`NULL`).

## 3. Хеширование refresh-токенов в БД

`RefreshToken.Token` сейчас хранится сырым — компрометация БД сразу отдаёт все активные сессии.

**Шаги:**
- В `TokenService`/`AuthController.IssueTokens`: генерировать случайный токен как раньше, но в БД (`RefreshToken.Token`) сохранять его SHA-256 хеш; сам сырой токен отдавать клиенту только в ответе (как сейчас).
- `RefreshTokenRepository.GetByToken`: принимать сырой токен от клиента, искать по хешу.
- Обновить существующий индекс `IX_RefreshTokens_Token` (из `AddAuthLookupIndexes`) — он останется валиден, так как индексируется просто колонка `Token`, независимо от того, что в ней хеш, а не сырое значение.
- Проверить вживую: `signin` → `refresh` с полученным токеном отрабатывает; прямой SQL-запрос к `RefreshTokens.Token` не показывает сырое значение.

**Сделано.** `ITokenService.HashRefreshToken(rawToken)` — SHA-256, `Convert.ToHexString`. `AuthController.IssueTokens` сохраняет хеш в `RefreshToken.Token`, клиенту в ответе по-прежнему уходит сырой токен. `Refresh`/`Revoke` хешируют токен из тела запроса перед вызовом `GetByToken` (сам репозиторий не менялся — как искал по точному совпадению колонки, так и ищет, просто теперь ему передают хеш вместо сырого значения). Миграция не потребовалась — тип колонки не изменился, только формат хранимого значения. Существующие до этого изменения refresh-токены в БД (сырые) естественным образом становятся невалидными — пользователям придётся перелогиниться, это ожидаемо, не баг.

Проверено вживую: `signin` → в БД `RefreshTokens.Token` — `3A5B6AFE...` (SHA-256 хеш, не сырое значение) → `refresh` с сырым токеном, который получил клиент, — `200`, новые токены выданы → `revoke` с сырым токеном — `204` → повторный `refresh` тем же (уже отозванным) токеном — `401`.

## 4. Reuse-detection на refresh-токенах

Ротация работает (`IsRevoked` при `/refresh`), но повторное предъявление уже отозванного токена просто `401`, без реакции.

**Шаги:**
- В `AuthController.Refresh`: если найденный (по хешу) токен помечен `IsRevoked = true` — это признак угона (кто-то использует уже потраченный токен). Отозвать **все** активные `RefreshToken` этого пользователя (`IsRevoked = true` массово).
- Залогировать инцидент (`ILogger`, `UserId`, `IP`) — задел под будущий пункт 8 (аудит), но не обязательно ждать его.
- Проверить вживую: `refresh` дважды одним и тем же токеном → второй раз все токены пользователя отозваны, следующий `refresh` валидным токеном тоже должен упасть.

**Сделано.** `IRefreshTokenRepository.RevokeAllForUser(userId)` — массово отзывает все ещё активные токены пользователя одним запросом. `AuthController.Refresh` теперь различает три случая явно: токен не найден → `401`; токен найден, но уже отозван (reuse) → `RevokeAllForUser` + `ILogger.LogWarning` (`UserId`, `RemoteIp`) + `401`; токен найден, не отозван, но просрочен → `401`. Раньше все три схлопывались в одно `if` без разбора.

Проверено вживую сценарием с двумя параллельными сессиями одного пользователя (токены A и B): `refresh` токеном A (ротация, A → revoked, выдан новый C) → повторное предъявление уже отозванного A → `401` + в логе `"Refresh token reuse detected for user ... — all active sessions revoked."` → проверка в БД: было 8 активных токенов, стало `0` → токен B (совершенно отдельная, ранее валидная сессия) тоже отклонён `401` — подтверждено, что угон одного токена гасит все сессии пользователя, не только скомпрометированную.
