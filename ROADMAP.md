# BusBooking — тех. стек и план до деплоя

## Технологический стек

### Backend

| Компонент           | Выбор                                                  | Почему                                                                                                                                                                 |
| ------------------- | ------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Фреймворк           | ASP.NET Core 8 Web API                                 | Уже есть готовая кодовая база (сущности, EF Core, контроллеры) — дешевле довести до ума, чем переписывать с нуля                                                       |
| ORM                 | Entity Framework Core 8                                | Уже используется, миграции частично написаны                                                                                                                           |
| База данных         | PostgreSQL                                             | Дешёвый/бесплатный managed-хостинг почти везде (Railway, Render, Fly.io, Supabase), в отличие от managed SQL Server                                                    |
| Аутентификация      | JWT (access + refresh токен), выдаётся API             | Фронтенд и бэкенд на разных доменах (Next.js на Vercel, API отдельно) — cookie-сессии между origin усложняют CORS/SameSite; JWT — стандарт для разделённой архитектуры |
| Хэширование паролей | `PasswordHasher<T>` (ASP.NET Core Identity) или BCrypt | Текущий голый SHA-256 без соли небезопасен                                                                                                                             |
| Валидация           | FluentValidation или DataAnnotations                   | Серверная валидация нужна независимо от клиентской — API публичный                                                                                                     |

### Frontend

| Компонент                 | Выбор                             | Почему                                                                                              |
| ------------------------- | --------------------------------- | --------------------------------------------------------------------------------------------------- |
| Фреймворк                 | Next.js (App Router) + TypeScript | SEO — реальная бизнес-задача, нужен SSR/SSG для публичных страниц                                   |
| Стилизация                | Tailwind CSS                      | Есть готовый визуальный дизайн (фиолетовый градиент, карточки) — быстро ложится на кастомный дизайн |
| Работа с API / кэш данных | TanStack Query                    | Поиск рейсов, доступность мест — данные нужно кэшировать и обновлять                                |
| Валидация форм            | Zod                               | TypeScript-нативная схема-валидация, переиспользуется на клиенте                                    |
| Карта маршрутов           | Leaflet + OpenStreetMap           | В дизайн-макете (`Map.jpg`) уже атрибуция Leaflet/OpenStreetMap — бесплатно, без API-ключей         |

### Инфраструктура и деплой

| Компонент                  | Выбор                                                | Почему                                                                                 |
| -------------------------- | ---------------------------------------------------- | -------------------------------------------------------------------------------------- |
| Контейнеризация            | Docker (multi-stage: `dotnet/sdk` → `dotnet/aspnet`) | Портируемость backend+DB независимо от хостинга                                        |
| Хостинг backend + Postgres | Fly.io / Railway / Render                            | Простой деплой контейнера + managed Postgres в одном месте                             |
| Хостинг frontend           | Vercel                                               | Next.js — их фреймворк, деплой без конфигурации, SSR/ISR/Edge из коробки               |
| CI/CD                      | GitHub Actions                                       | build+test backend → Docker-образ → деплой; Vercel сам подхватывает пуши для фронтенда |

### Тестирование (сейчас отсутствует полностью)

### Фаза 2 — Безопасность

Проект теперь идёт в прод для реального клиента — пункты ниже обязательны, не опциональны.

- [+] Заменить SHA-256 на `PasswordHasher<T>`/BCrypt для паролей

- [+] Rate limiting — `AddRateLimiter` в `Program.cs`, `GlobalLimiter` со sliding window (100 запросов/мин на IP), `RejectionStatusCode = 429`. Реально протестировано burst-запросами (99×200 + 6×429 на границе лимита)
- [+] **Уточнение к rate limiting** — добавлена именованная политика `"auth"` (`AddPolicy`, sliding window, 5/мин на IP) поверх глобального лимитера, навешена через `[EnableRateLimiting("auth")]` на `POST /api/auth/signin`/`signup`. Проверено вживую: 5 разрешённых попыток + `429` на 6-й и 7-й, остальные эндпоинты не затронуты. Подробности — `SECURITY_PHASE2_PLAN.md`, пункт 1.
- [+] **Account lockout после N неудачных попыток входа** — `User.FailedLoginAttempts`/`LockoutEnd` (миграция `AddAccountLockout`), `AuthController.SignIn` проверяет блокировку до `VerifyPassword`, инкремент при неверном пароле, `LockoutEnd = +15 мин` при достижении 5 попыток, сброс при успехе. Возвращает `423 Locked`, не `401` — клиент может отличить «неверный пароль» от «аккаунт временно заблокирован». Проверено вживую полным циклом (5 неудачных → верный пароль всё равно `423` → после истечения блокировки — успешный вход со сбросом счётчика). Подробности — `SECURITY_PHASE2_PLAN.md`, пункт 2.
- [+] **Refresh-токены хранятся в БД в открытом виде** — `ITokenService.HashRefreshToken` (SHA-256), `RefreshToken.Token` теперь хранит хеш, не сырое значение; клиент по-прежнему получает сырой токен, `Refresh`/`Revoke` хешируют его перед поиском в БД. Проверено вживую: `signin` → в БД хеш, не сырое значение → `refresh`/`revoke` сырым токеном работают корректно → повторный `refresh` после `revoke` — `401`. Подробности — `SECURITY_PHASE2_PLAN.md`, пункт 3.
- [+] **Reuse-detection на refresh-токенах** — `AuthController.Refresh` теперь при обнаружении уже отозванного токена вызывает `IRefreshTokenRepository.RevokeAllForUser` (гасит все активные сессии пользователя, не только скомпрометированную) и логирует инцидент через `ILogger`. Проверено вживую: угон одного из двух параллельных токенов пользователя корректно обнуляет обе сессии. Подробности — `SECURITY_PHASE2_PLAN.md`, пункт 4.
- [+] **CORS вынесен в конфигурацию** — `Cors:AllowedOrigins` (`string[]`) в `appsettings.json` (дефолт `http://localhost:3000` для дева), `Program.cs` падает на старте с понятной ошибкой, если секция не задана. Переопределяется стандартным способом ASP.NET Core через переменные окружения (`Cors__AllowedOrigins__0=https://prod-domain`, без изменений в коде). Прод-домен фронтенда добавить перед деплоем (Фаза 8). Проверено вживую: дефолт (`localhost:3000`) — preflight с `Origin: http://evil.com` не получает `Access-Control-Allow-Origin`; с `Cors__AllowedOrigins__0=https://example-prod.com` — новый origin разрешён, старый `localhost:3000` корректно отклонён (список полностью заменяется, не расширяется).
- [+] **Security response headers** — инлайн-middleware в начале пайплайна (`Program.cs`, сразу после `app.Build()`) добавляет `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: no-referrer` на все ответы, и `Content-Security-Policy: default-src 'none'; frame-ancestors 'none'; base-uri 'none'` на всё, кроме `/swagger/*` (Swagger UI, только в деве, рендерит HTML с инлайн-скриптами/стилями — обычному JSON API эти заголовки не нужны, отдельный пакет не понадобился). HSTS уже был включён вне Development. Проверено вживую: `/health` и `/api/routes` отдают все четыре заголовка, `/swagger/index.html` отдаёт `200` и реально рендерит `swagger-ui` без CSP-заголовка.
- [+] **Проверка пароля через Have I Been Pwned** — `IPasswordBreachChecker`/`PasswordBreachChecker` (k-anonymity: SHA-1 хеш пароля, на API уходит только первые 5 hex-символов, полный хеш и пароль никогда не покидают процесс), вызывается в `AuthController.SignUp` до создания пользователя, `400` с понятным сообщением при совпадении. `SignUp` стал `async` ради этого HTTP-вызова (единственный async-метод в контроллере — остальной слой данных синхронный, см. Фазу 10.6). HTTP-клиент — `static readonly HttpClient` (ServiceManager per-request, полю на инстансе грозило бы исчерпание сокетов). Fail-open при недоступности HIBP API (не блокировать регистрацию из-за стороннего аутейджа) — осознанный компромисс, не баг. Произвольные правила сложности (upper/digit/symbol) сознательно не добавлены — низкая реальная защита. Проверено вживую: `signup` с `password123` (заведомо в утечках) → `400`, с уникальным сгенерированным паролем → `200` + токены.
- [ ] **Нет аудит-лога админ-действий** — `TripsController` (Create/Update/Delete) и `UsersController` (все admin-эндпоинты) не логируют, кто и когда выполнил действие. Минимум — `ILogger`-запись с `UserId`/действием/объектом; в идеале — отдельная таблица `AuditLog` для реальной истории изменений.
- [ ] **JWT signing key — единый статичный ключ, без ротации** (`Services/TokenService.cs:22`, `configuration["Jwt:SigningKey"]`) — компрометация ключа делает недействительными все выданные токены разом без плавного перехода. Для MVP не блокирует, но стоит заложить процедуру ротации (двойная валидация по старому+новому ключу на переходный период) до реального прод-инцидента.

### Фаза 5 — Бизнес-функционал

Без этого MVP не является рабочим продуктом для реальной автобусной компании — сейчас можно "забронировать" место без единого подтверждения оплаты, а рейсы создаются вручную по одному.

- [+] **Оплата** — Stripe Checkout подключён: `PaymentsController` (`POST /api/payments/checkout/{orderId}` создаёт сессию и сохраняет `Order.StripeSessionId`; `POST /api/payments/webhook`, с проверкой `Stripe-Signature`, подтверждает заказ по событию `checkout.session.completed`). Проверено вживую: hold места → checkout (502 с чистым сообщением без реального ключа — ожидаемо в дев-окружении) → webhook с поддельной подписью корректно отклоняется (`400`). Реальный `Stripe:SecretKey`/`WebhookSecret` — через `dotnet user-secrets`, не в `appsettings.json`. Фронтенд-виджет оплаты — отдельный пункт в Фазе 7.
- [+] **Статус заказа** — `Order.Status` (`OrderStatus`: `PendingPayment`/`Confirmed`/`Cancelled`/`Expired`; `Refunded` пока не добавлен — не нужен без реализованного возврата, см. пункт «Отмена бронирования» ниже), миграция `AddOrderStatusAndSeatHold` применена.
- [+] **Тайм-аут удержания места (seat hold)** — `SeatDetail.ReservedUntil`, `BookSeat` теперь только резервирует место на 15 минут (`HoldSeat`), а не занимает его навсегда; `IsOccupied` выставляется только вебхуком после реального платежа (`ConfirmSeat`). Просроченные holds освобождаются лениво при каждом `GET /api/seats/trip/{tripId}` (`ReleaseExpiredHolds`), связанный `Order` в статусе `PendingPayment` переводится в `Expired`. Проверено вживую: второй пользователь получает `409` на уже удерживаемое место. Фоновая задача для проактивной очистки (а не только при следующем чтении) — не сделана, для MVP-масштаба достаточно lazy-подхода.
- [+] **Баг: место навсегда недоступно после первого истёкшего hold'а** — обнаружено вручную при проверке Фазы 10.6. Исправлено в `SeatsController.ExpireStaleHolds`: истёкшие `OrderSeat`-записи теперь удаляются (`_unitOfWork.OrderSeat.RemoveRange(...)`) вместе с переводом заказа в `Expired`, а не оставляются висеть, блокируя уникальный индекс `OrderSeat.SeatDetailId`. Проверено вживую: hold → принудительно состарен → место освобождено → **повторное бронирование того же места тем же пользователем прошло успешно** (регресс-тест бага). Подробности — `BOOKING_FLOW_PLAN.md`, пункт 0.
- [+] **Отмена бронирования и возврат** — `POST /api/bookings/{orderId}/cancel` (клиент, своя бронь, только пока `Trip.DepartureDate` в будущем) и `POST /api/bookings/{orderId}/admin-cancel` (`[Authorize(Roles = Admin)]`, обязательная причина, без дедлайна) — оба поверх общей `CancelOrder`: `Confirmed` → возврат через Stripe (`IPaymentService.RefundPayment`) → `Refunded`; `PendingPayment` → сразу `Cancelled`; место освобождается тем же способом, что и при истечении hold'а (Фаза 5 выше). `Order.StripePaymentIntentId` теперь сохраняется в вебхуке для возможности возврата. Фронтенд: кнопка "Cancel booking" на `/bookings` (только для отменяемых статусов), `OrderSummaryVM.Status` добавлен. Проверено вживую 9 сценариями (успешная отмена + освобождение места + регресс-тест повторного бронирования, дедлайн, повторная отмена, admin-cancel с/без причины, отказ не-админу, неудачный возврат с сохранением консистентного статуса, отмена без `PaymentIntentId`) — подробности в `CANCELLATION_REFUND_PLAN.md`.

Доступность места считается не булевым `IsOccupied`, а проверкой пересечения отрезков `[StopOrder, StopOrder]` существующих бронирований на то же место того же рейса — одно место можно продать нескольким пассажирам на непересекающихся участках одного рейса. Цена отрезка = разница `PriceFromStart` конечной и начальной остановки. Проверено вживую: маршрут Sofia→Berlin на `/routes` показывает изгибающуюся через реальные города линию вместо прямой.
- [ ] **Билет туда-обратно со скидкой** — `Order.ReturnTripId` (nullable) + `RouteInfo.RoundTripDiscountPercent`; `OrderSeat.TripId` уже поддерживает разные рейсы в рамках одного заказа, менять не нужно. Клиент обязан выбрать оба рейса (туда и обратно) с конкретной датой/временем при оформлении — не открытую дату возврата.
- [ ] **Водители** — новая сущность `Driver` (имя, контакты, категория прав) и связь `Trip → Driver`. Актуально именно потому, что компания владеет автопарком сама — нужно на что-то ставить расписание смен, а не только автобусы. Не блокирует MVP-флоу бронирования — можно назначать водителей вручную/вне системы, пока фича не реализована.
- [ ] **Детская цена** — скидка на билет для детей. Затрагивает многоместное бронирование (`ROADMAP.md`/`BOOKING_FLOW_PLAN.md`, «многоместное бронирование»): тип пассажира нужен **на каждое место отдельно**, не на заказ целиком (семья может бронировать места и взрослым, и детям в одном заказе). Нужны: `RouteInfo.ChildDiscountPercent` (по аналогии с уже описанным `RoundTripDiscountPercent` выше — единая базовая цена + процент скидки, а не отдельная детская цена на каждый маршрут вручную), `OrderSeat.PassengerType` (`Adult`/`Child`, enum). `BookSeatRequestVM.SeatIds: List<Guid>` придётся заменить на список пар `{ SeatId, PassengerType }`, `Order.TotalPrice` считать посадочно, а не `trip.Price × количество мест`, как сейчас. На фронтенде — переключатель "взрослый/ребёнок" на каждое выбранное место в `/trips/[tripId]/seats`, до подтверждения бронирования. Нужно решить возрастной порог "ребёнка" (детали ценообразования — вне рамок разработки, уточнить у владельца бизнеса).
