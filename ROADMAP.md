# BusBooking — тех. стек и план до деплоя

## Технологический стек

### Backend

| Компонент           | Выбор                                                  | Почему                                                                                                                                                                 |
| ------------------- | ------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Фреймворк           | ASP.NET Core 8 Web API                                 | Уже есть готовая кодовая база (сущности, EF Core, контроллеры) — дешевле довести до ума, чем переписывать с нуля                                                       |
| ORM                 | Entity Framework Core 8                                | Уже используется, миграции частично написаны                                                                                                                           |
| База данных         | PostgreSQL                                             | Дешёвый/бесплатный managed-хостинг почти везде (Railway, Render, Fly.io, Supabase), в отличие от managed SQL Server                                                    |
| Аутентификация      | JWT (access + refresh токен), выдаётся API             | Фронтенд и бэкенд на разных доменах (Next.js на Vercel, API отдельно) — cookie-сессии между origin усложняют CORS/SameSite; JWT — стандарт для разделённой архитектуры |
| Хэширование паролей | `PasswordHasher<T>` (ASP.NET Core Identity) или BCrypt | Текущий голый SHA-256 без соли небезопасен                                                                                                                             |
| Валидация           | FluentValidation или DataAnnotations                   | Серверная валидация нужна независимо от клиентской — API публичный                                                                                                     |

### Frontend

| Компонент                 | Выбор                             | Почему                                                                                              |
| ------------------------- | --------------------------------- | --------------------------------------------------------------------------------------------------- |
| Фреймворк                 | Next.js (App Router) + TypeScript | SEO — реальная бизнес-задача, нужен SSR/SSG для публичных страниц                                   |
| Стилизация                | Tailwind CSS                      | Есть готовый визуальный дизайн (фиолетовый градиент, карточки) — быстро ложится на кастомный дизайн |
| Работа с API / кэш данных | TanStack Query                    | Поиск рейсов, доступность мест — данные нужно кэшировать и обновлять                                |
| Валидация форм            | Zod                               | TypeScript-нативная схема-валидация, переиспользуется на клиенте                                    |
| Карта маршрутов           | Leaflet + OpenStreetMap           | В дизайн-макете (`Map.jpg`) уже атрибуция Leaflet/OpenStreetMap — бесплатно, без API-ключей         |

### Инфраструктура и деплой

| Компонент                  | Выбор                                                | Почему                                                                                 |
| -------------------------- | ---------------------------------------------------- | -------------------------------------------------------------------------------------- |
| Контейнеризация            | Docker (multi-stage: `dotnet/sdk` → `dotnet/aspnet`) | Портируемость backend+DB независимо от хостинга                                        |
| Хостинг backend + Postgres | Fly.io / Railway / Render                            | Простой деплой контейнера + managed Postgres в одном месте                             |
| Хостинг frontend           | Vercel                                               | Next.js — их фреймворк, деплой без конфигурации, SSR/ISR/Edge из коробки               |
| CI/CD                      | GitHub Actions                                       | build+test backend → Docker-образ → деплой; Vercel сам подхватывает пуши для фронтенда |

### Фаза 5 — Бизнес-функционал

Без этого MVP не является рабочим продуктом для реальной автобусной компании — сейчас можно "забронировать" место без единого подтверждения оплаты, а рейсы создаются вручную по одному.

- [ ] **Оплата** — интеграция с платёжным провайдером (Stripe Checkout или локальный эквайринг). Бронирование не должно считаться подтверждённым, пока оплата не прошла. Сейчас `Order` создаётся и место помечается занятым сразу при выборе, без проверки денег вообще.
- [ ] **Статус заказа** — у `Order` сейчас нет поля `Status`. Добавить enum (`PendingPayment`/`Confirmed`/`Cancelled`/`Refunded`) — без него не построить ни оплату, ни отмену, ни отчётность.
- [ ] **Тайм-аут удержания места (seat hold)** — при выборе места оно должно резервироваться на ограниченное время (например, 10-15 минут) до завершения оплаты, а не занимать место навсегда. Если оплата не пришла в срок — место автоматически освобождается (фоновая задача или проверка TTL при каждом обращении к местам рейса).
- [ ] **Отмена бронирования и возврат** — action для клиента (отменить свою бронь до дедлайна) и для админа (отменить чужую с указанием причины), завязанные на поле `Status` выше.
- [ ] **Автогенерация рейсов из `RouteInfo`** — маршрут уже хранит регулярность (`DepartureDay`/`ReturnDay`, например "каждый понедельник Sofia→Berlin"), но `Trip` сейчас создаётся вручную по одному через форму. Нужна генерация рейсов на N недель вперёд (фоновая задача или кнопка "сгенерировать рейсы на месяц" в админке).
- [ ] **E-mail подтверждение бронирования** — после оплаты отправлять клиенту письмо с деталями рейса, местом и кодом заказа. Сейчас email вообще нигде не отправляется, инфраструктуры для писем в проекте нет.
- [ ] **Водители** — новая сущность `Driver` (имя, контакты, категория прав) и связь `Trip → Driver`. Актуально именно потому, что компания владеет автопарком сама — нужно на что-то ставить расписание смен, а не только автобусы.
- [ ] **Напоминание перед отправлением** — email/SMS за N часов до рейса (зависит от e-mail-интеграции выше).
- [ ] **Промежуточные остановки и бронирование по участкам** — новая сущность `RouteStop` (город, `StopOrder`, `PriceFromStart`), у `Order` — поля `FromStopId`/`ToStopId`. Доступность места считается не булевым `IsOccupied`, а проверкой пересечения отрезков `[StopOrder, StopOrder]` существующих бронирований на то же место того же рейса — одно место можно продать нескольким пассажирам на непересекающихся участках одного рейса. Цена отрезка = разница `PriceFromStart` конечной и начальной остановки.

**Опционально (можно отложить после запуска MVP):**

- [ ] **Билет туда-обратно со скидкой** — `Order.ReturnTripId` (nullable) + `RouteInfo.RoundTripDiscountPercent`; `OrderSeat.TripId` уже поддерживает разные рейсы в рамках одного заказа, менять не нужно. Клиент обязан выбрать оба рейса (туда и обратно) с конкретной датой/временем при оформлении — не открытую дату возврата.
